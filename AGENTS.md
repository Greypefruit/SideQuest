# AGENTS.md — SideQuest

## 1. Назначение документа

Этот документ задает **правила работы Codex внутри проекта SideQuest**.

Он определяет:

- как читать проектную документацию
- как интерпретировать задачи
- как вносить изменения в кодовую базу
- как избегать лишних изменений и переписывания уже работающего кода
- как проверять результат
- как отчитываться по выполненной задаче

Этот документ **не является** источником продуктовых требований, модели данных, ролевой модели, порядка этапов или технического стека.

---

## 2. Границы документа

`AGENTS.md` регулирует только:

- рабочий процесс
- дисциплину изменений
- правила интерпретации задач
- правила внесения изменений
- правила проверки результата
- правила отчетности

`AGENTS.md` **не должен** дублировать:

- продуктовую логику из `PRD.md`
- сущности и связи из `DOMAIN_MODEL.md`
- роли и права из `RBAC.md`
- порядок реализации из `TASKS.md`
- стек и технические ограничения из `TECH_STACK.md`
- правила валидации из `VALIDATION_RULES.md`

Если нужно понять продуктовую логику, модель данных, роли, этапы работ, стек или валидации, нужно читать соответствующий документ, а не додумывать их по `AGENTS.md`.

---

## 3. Точка входа в документацию

Перед началом любой задачи сначала нужно открыть **`DOC_INDEX.md`**.

`DOC_INDEX.md` является навигационным документом проекта и определяет:

- какие документы существуют в проекте
- за что отвечает каждый документ
- в каком порядке их читать
- какой документ является источником истины для конкретного типа вопроса

Codex не должен выбирать документы по догадке, если структура уже описана в `DOC_INDEX.md`.

---

## 4. Иерархия документов

При выполнении любой задачи использовать документы в следующем порядке:

1. `DOC_INDEX.md`
2. `PRD.md`
3. `DOMAIN_MODEL.md`
4. `RBAC.md`
5. `TASKS.md`
6. `TECH_STACK.md`
7. `VALIDATION_RULES.md`
8. `AGENTS.md`

### Правило при конфликте

Если между документами есть конфликт, приоритет выше у документа, который стоит выше в списке.

Это означает:

- `DOC_INDEX.md` задает только навигацию и не заменяет документы-источники истины
- `PRD.md` имеет приоритет над всеми содержательными документами ниже
- `DOMAIN_MODEL.md` имеет приоритет над `RBAC.md`, `TASKS.md`, `TECH_STACK.md`, `VALIDATION_RULES.md`, `AGENTS.md`
- `RBAC.md` имеет приоритет над `TASKS.md`, `TECH_STACK.md`, `VALIDATION_RULES.md`, `AGENTS.md`
- `TASKS.md` имеет приоритет над `TECH_STACK.md`, `VALIDATION_RULES.md`, `AGENTS.md`
- `TECH_STACK.md` имеет приоритет над `VALIDATION_RULES.md` и `AGENTS.md` в вопросах стека и инфраструктуры
- `VALIDATION_RULES.md` имеет приоритет над `AGENTS.md` в вопросах негативных сценариев и проверок

Если возникает конфликт, нельзя молча выбирать удобную интерпретацию. Нужно следовать документу с более высоким приоритетом.

---

## 5. Обязательный порядок работы над любой задачей

Перед началом любой реализации обязательно:

1. прочитать текущую задачу пользователя
2. прочитать `DOC_INDEX.md`
3. прочитать остальные документы в порядке, указанном в `DOC_INDEX.md`
4. определить, какой именно этап проекта сейчас реализуется
5. определить точные границы текущей задачи
6. определить, какие файлы относятся к задаче, а какие — нет

Нельзя начинать реализацию, опираясь только на локальный контекст текущего файла.

---

## 6. Правило узкой интерпретации задачи

Любую задачу нужно трактовать **максимально узко и буквально**.

Это означает:

- реализовывать только то, что прямо входит в текущую задачу
- не добавлять “полезные улучшения” без запроса
- не расширять scope самостоятельно
- не додумывать новые функции, если они не зафиксированы в документах
- не реализовывать элементы следующих этапов “заодно”

Если задача — сделать страницу рейтинга, нужно сделать страницу рейтинга.  
Не нужно “заодно” добавлять историю рейтинга, графики, фильтры, аналитику или альтернативные представления, если этого нет в текущей задаче.

## Правило входа и доступа Release 1

При интерпретации задач Release 1 Codex обязан считать, что вход в приложение выполняется по `email + одноразовый код подтверждения`.

Текущая продуктовая модель Release 1:

- пользователь может использовать любой корректный email
- email не ограничивается корпоративными доменами
- профиль пользователя создается при первом успешном входе
- новый пользователь автоматически получает роль `player`
- доступ пользователя может быть отозван через деактивацию в административном разделе

### Что нельзя делать

Codex не должен самовольно:

- возвращать ограничение входа только по корпоративным доменам
- вводить whitelist email без явного запроса
- вводить invite-only механику без явного запроса
- использовать в UI тексты про рабочую или корпоративную почту, если вход разрешен по любому email
- строить auth flow вокруг предварительного одобрения email, если это не зафиксировано в текущем `PRD.md`

### Как правильно интерпретировать Release 1

Правильная интерпретация Release 1:

- приложение остается внутренним MVP
- вход выполняется по `email + OTP`
- любой корректный email допустим
- при первом успешном входе создается профиль
- доступ деактивированного пользователя блокируется через `is_active`, а не через домен email

## Правило activity scope для Release 1

При интерпретации задач Release 1 нельзя считать основные пользовательские экраны глобальными экранами "по всей платформе", если в документации явно не указано иное.

Даже если в Release 1 пользовательский интерфейс не показывает отдельный фильтр или переключатель активности, Codex обязан учитывать, что основные данные экранов работают **в контексте текущей активности**.

В Release 1 текущая активность фиксирована и равна:

- `table_tennis`
- **Настольный теннис**

### Что это означает для интерпретации задач

Если в задаче встречаются экраны или блоки:

- `Рейтинг`
- `Последние матчи`
- `Матчи`
- `Турниры`
- `Страница турнира`
- `Профиль`

нужно трактовать их как activity-scoped экраны и выборки, а не как глобальные экраны по всем активностям сразу.

При реализации auth-экранов Codex обязан использовать тексты и состояния, соответствующие текущей модели доступа Release 1.

Если вход разрешен по любому email, интерфейс не должен содержать формулировки:
- "рабочая почта"
- "корпоративная почта"
- "доступ только для корпоративного домена"

если это прямо не требуется текущим `PRD.md`.

### Что нельзя делать

Codex не должен самовольно:

- строить глобальный рейтинг по всем активностям
- смешивать матчи разных активностей в одном пользовательском списке
- смешивать турниры разных активностей в одном пользовательском списке
- писать выборки без activity scope для экранов, которые в Release 1 относятся к текущей активности
- интерпретировать отсутствие activity selector в UI как разрешение смешивать данные разных активностей

### Как правильно интерпретировать Release 1

Правильная интерпретация Release 1:

- UI работает только для настольного тенниса
- доменная модель остается расширяемой для нескольких активностей
- пользовательские выборки уже в Release 1 должны быть привязаны к текущей активности
- в будущем выбор активности должен добавляться поверх уже существующего activity-scoped подхода, а не через переписывание экранов с нуля

## 7. Правило минимального изменения

Любая задача должна решаться **минимальным достаточным набором изменений**.

Нужно:

- менять только те файлы, которые реально относятся к задаче
- вносить только те изменения, которые нужны для результата
- сохранять уже работающий код, если его не требуется менять
- расширять систему, а не переписывать ее без необходимости

Нельзя:

- трогать unrelated файлы
- реорганизовывать проект “заодно”
- переписывать существующий модуль только потому, что можно сделать “красивее”
- менять названия базовых сущностей без явного запроса

---

## 8. Правило отсутствия скрытого рефакторинга

Codex не должен делать скрытый рефакторинг.

Под скрытым рефакторингом понимается ситуация, когда под видом выполнения одной задачи дополнительно происходит:

- изменение архитектуры
- массовое переименование
- перенос логики между слоями
- переписывание уже работающих модулей
- переразметка маршрутов
- замена уже работающего подхода на новый
- изменение большого числа файлов без прямой необходимости

### Разрешено

Разрешены только такие рефакторинги, которые:

- непосредственно нужны для выполнения текущей задачи
- малы по объему
- не меняют продуктовую логику
- не ломают существующие контракты

### Не разрешено

Нельзя:

- делать большой рефакторинг без отдельного запроса
- менять структуру проекта “на вырост”
- переписывать модуль ради “идеальной архитектуры”
- менять существующие контракты без явного решения

---

## 9. Правило сохранения контрактов

Если в проекте уже существует рабочий контракт, его нельзя менять без явного запроса.

Под контрактом понимаются:

- названия базовых сущностей
- структура маршрутов
- имена экспортов
- интерфейсы функций
- структура данных, на которую уже опирается код
- рабочая логика уже согласованного флоу

Если контракт необходимо изменить, это должно быть:

- прямо нужно для текущей задачи
- ограничено по объему
- явно объяснено в отчете

---

## 10. Правило дисциплины файлов

Перед внесением изменений нужно определить:

- какие файлы разрешено менять
- какие файлы не нужно менять
- какие подсистемы не относятся к текущей задаче

### Общий принцип

Если файл не нужен для решения задачи, его не нужно изменять.

### Особенно осторожно относиться к

- конфигурационным файлам
- инфраструктурным файлам
- файлам подключения к БД
- auth-механике
- маршрутам
- общим layout-компонентам
- существующим миграциям
- seed-данным

Если изменение этих файлов не требуется напрямую, их нельзя трогать.

---

## 11. Правило поэтапной реализации

Проект должен развиваться **строго по шагам**, а не скачками.

Codex обязан:

- следовать этапам из `TASKS.md`
- не перепрыгивать через незавершенные этапы
- не строить функции из следующих задач раньше времени
- не пытаться закрыть весь Release 1 одной реализацией

Если текущая задача зависит от предыдущей, сначала должна быть доведена до рабочего состояния предыдущая задача.

---

## 12. Правило осторожного отношения к данным и инфраструктуре

Любые изменения, связанные с данными, схемой, миграциями, конфигурацией или инфраструктурой, требуют повышенной осторожности.

Нельзя без явного запроса:

- удалять рабочие настройки
- ломать существующее подключение
- менять рабочие переменные окружения
- удалять существующие таблицы
- выполнять destructive changes без необходимости
- удалять или переписывать существующие миграции

Если изменение затрагивает данные или инфраструктуру, нужно действовать минимально и предсказуемо.

---

## 13. Правило локальности задачи

Каждая задача должна заканчиваться законченным и проверяемым результатом в пределах своей области.

Хорошая задача:

- ограничена по объему
- имеет понятный результат
- не требует перепроектирования всего приложения
- может быть проверена отдельно

Плохая задача:

- затрагивает слишком много подсистем сразу
- меняет модель, UI, auth и инфраструктуру одновременно
- не имеет четких границ завершения

Codex должен удерживать локальность задачи и не размывать ее.

---

## 14. Правило проверки предположений

Если что-то не зафиксировано в документах, нельзя автоматически считать это допустимым.

Нельзя:

- придумывать новые сущности
- вводить новые роли
- расширять продуктовый flow
- добавлять новые статусы
- вводить новые экраны
- менять бизнес-правила

только потому, что это “может пригодиться”.

Если правило не зафиксировано, нужно следовать текущему scope, а не предполагать будущее расширение.

Если текущие документы явно фиксируют модель входа и доступа, Codex не должен возвращать прежние ограничения "по памяти" или на основании типовых паттернов.

В частности, если `PRD.md` не требует ограничивать вход корпоративными доменами, Codex не должен добавлять это ограничение самостоятельно.

---

## 15. Правило интерпретации экранов и UX

При реализации экранов нужно опираться на `PRD.md` как на источник истины по UX и пользовательским сценариям.

Это означает:

- не придумывать новые экраны вне scope
- не добавлять лишние действия на экран
- учитывать зафиксированные в `PRD.md` требования к русскоязычному интерфейсу
- учитывать зафиксированные в `PRD.md` требования к адаптивности и mobile-priority сценариям
- не проектировать экран только под desktop, если в `PRD.md` мобильный сценарий является приоритетным

---

## 16. Правило объяснимости изменений

После выполнения задачи нужно кратко и честно объяснить результат.

Отчет должен включать:

1. что было сделано
2. какие файлы были изменены
3. какие проверки были выполнены
4. какие ограничения соблюдались
5. что не было сделано, если это оставалось вне scope

### В отчете нельзя

- преувеличивать степень готовности
- утверждать, что реализованы функции, которых нет
- описывать большой объем работы, если сделаны только частичные изменения
- скрывать, что часть задачи не была закрыта

---

## 17. Правило обязательной валидации

Перед завершением задачи нужно выполнить минимально разумную проверку результата.

Нужно проверять то, что относится к текущей задаче.

Примеры проверок:

- проект собирается
- страница открывается
- нужный сценарий работает
- данные читаются или сохраняются
- ограничения доступа соблюдаются
- существующий рабочий функционал не сломан

Нельзя завершать задачу без попытки проверки результата.

---

## 18. Правило отсутствия ложной универсальности

SideQuest должен быть расширяемым, но Codex не должен строить слишком общую систему раньше времени.

Это означает:

- не создавать универсальные механизмы без практической пользы текущему этапу
- не вводить дополнительный abstraction layer без явной нужды
- не строить “framework внутри проекта”
- не превращать MVP в платформу-конструктор

Расширяемость допустима только тогда, когда она не усложняет Release 1.

---

## 19. Правило аккуратной эволюции проекта

Новая функциональность должна:

- встраиваться в существующий каркас
- использовать уже согласованные сущности
- использовать уже существующие документы как источник истины
- расширять текущую архитектуру без ее разрушения

Если новое решение требует ломать старое, это уже не “малое наращивание”, а изменение архитектуры, и оно требует отдельного явного решения.

---

## 20. Правило отсутствия самовольного расширения scope

Codex не должен сам добавлять:

- новые страницы
- новые режимы
- новые типы данных
- новые настройки
- новые административные инструменты
- дополнительные UX-сценарии
- дополнительные поля в формах
- новые API-endpoints
- дополнительные domain objects

если этого нет в текущей задаче или документации.

---

## 21. Обязательный шаблон поведения перед реализацией

Перед началом работы над задачей Codex должен ответить на вопросы:

1. Какой этап сейчас реализуется?
2. Что именно входит в scope этой задачи?
3. Что в эту задачу не входит?
4. Какие документы являются источником истины?
5. Какие файлы действительно нужно менять?
6. Какие контракты нельзя ломать?
7. Какие проверки нужно выполнить после изменений?

Если на один из этих вопросов нет ответа, нельзя расширять задачу по собственному усмотрению.

---

Нельзя без явного запроса:

- менять продуктовые требования
- менять доменную модель
- менять RBAC-модель
- менять стек проекта
- менять инфраструктурный способ запуска

- делать массовый рефакторинг

- трогать unrelated файлы
утверждать, что задача завершена, если это не так
самовольно ограничивать вход только корпоративными доменами
