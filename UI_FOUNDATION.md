# UI_FOUNDATION.md — SideQuest Release 1

## 1. Назначение документа

Этот документ фиксирует **визуальную основу интерфейса** для проекта **SideQuest Release 1**.

Документ нужен, чтобы:
- не переизобретать визуальный стиль на каждом новом экране
- удерживать единый UI-язык между `auth`, `home`, `tournaments`, `profile`, `ranking`, `matches`
- дать Codex и разработке понятные правила для layout, navigation, компонентов и состояний
- снизить количество случайных дизайн-решений по ходу реализации

Этот документ не заменяет:
- `PRD.md` — продуктовую логику
- `DOMAIN_MODEL.md` — доменную модель
- `RBAC.md` — роли и доступы
- `VALIDATION_RULES.md` — валидацию и защиту от невалидных состояний
- `AGENTS.md` — правила работы Codex

Если возникает конфликт:
- `PRD.md` важнее визуальных предпочтений
- продуктовый scope важнее визуальных идей
- уже утвержденные экраны важнее новых спонтанных решений

---

## 2. Общий визуальный принцип

### 2.1 Характер интерфейса

SideQuest Release 1 должен выглядеть как:

- спокойный внутренний рабочий инструмент
- аккуратный internal SaaS
- понятное приложение для ежедневного использования
- продукт без спортивного пафоса и без consumer-app подачи

SideQuest Release 1 не должен выглядеть как:
- спортивное медиа-приложение
- consumer event platform
- аналитический dashboard
- “игровой” или “соревновательный” интерфейс
- тяжелая административная система с множеством панелей

### 2.2 Базовый принцип композиции

Интерфейс строится вокруг простых и понятных смысловых блоков:
- один экран = один понятный сценарий
- одна секция = один смысловой блок
- один главный action = один главный фокус внимания

Основной подход:
- минимум декора
- максимум ясности
- воздух важнее украшений
- данные важнее эффектов
- действия важнее декоративных акцентов

### 2.3 Release 1 — это не финальный “маркетинговый” продукт

Release 1 должен быть:
- чистым
- строгим
- дружелюбным
- стабильным
- предсказуемым

Release 1 не должен:
- “показывать вау”
- имитировать consumer growth product
- компенсировать нехватку продукта визуальными эффектами

## 3. Визуальная основа

## 3.1 Фон

Базовый фон приложения:
- светлый
- холодно-нейтральный
- почти бело-серый

Фон не должен быть:
- чисто черным
- теплым бежевым
- декоративным
- текстурным
- градиентным

## 3.2 Основные поверхности

Основные поверхности интерфейса:
- белые карточки
- белые формы
- белые content-блоки
- белые секции поверх светлого фона

Белые поверхности — основная модель UI Release 1.

## 3.3 Акцентный цвет

В Release 1 используется **один основной синий акцент**.

Синий акцент применяется для:
- primary buttons
- активного раздела навигации
- ссылок
- focus / selected state
- важных значений
- статуса `Идет`
- мягкого выделения активного или победившего объекта, если это действительно нужно

Синий не должен использоваться:
- как декоративный шум
- как массовая заливка всех карточек
- как замена иерархии layout

## 3.4 Дополнительные статусные цвета

Допустимый набор:
- `Идет` → синий
- `Завершен` → зеленый
- `Черновик` → нейтральный серый
- `Отменен` → нейтральный приглушенный
- ошибка / destructive state → красный

Статусы должны отображаться компактно и спокойно.

## 4. Типографика

## 4.1 Общий характер типографики

Типографика должна быть:
- спокойной
- строгой
- читабельной
- без декоративности

Не использовать:
- нарочито спортивные шрифтовые приемы
- чрезмерный uppercase
- большие display-headlines ради эффекта
- декоративные подписи

## 4.2 Иерархия текста

В Release 1 достаточно следующей иерархии:

### Page title
Главный заголовок страницы.

Используется для:
- `Главная`
- `Матчи`
- `Турниры`
- `Рейтинг`
- `Профиль`
- detail pages типа `Турнир`

### Section title
Заголовок секции внутри страницы.

Используется для:
- `Участники`
- `Сетка турнира`
- `Последние матчи`
- `Победитель`
- `Топ игроков`

### Card title / block title
Локальный заголовок карточки или смыслового блока.

### Body text
Основной текст интерфейса.

### Secondary text
Подписи, пояснения, дата/время, вторичный статусный текст.

### Label text
Подписи полей формы и маленькие системные labels.

## 4.3 Правила использования

- у страницы один главный заголовок
- у секции один заголовок второго уровня
- не использовать слишком много уровней подзаголовков
- вторичный текст всегда должен быть заметно спокойнее основного
- серый текст используется как вспомогательный, а не как основной носитель важных смыслов

## 5. Layout и spacing

## 5.1 Общие правила layout

Layout Release 1 должен быть:
- mobile-first
- одноколоночным на mobile
- расширяемым на desktop
- простым и повторяемым

Нельзя:
- строить экран как мозаичный dashboard
- перегружать страницу равносильными блоками
- ставить слишком много визуально одинаково важных секций подряд

## 5.2 Горизонтальные отступы

На mobile у всех экранов должен быть единый горизонтальный gutter.

Все корневые экраны должны визуально ощущаться частью одной системы:
- одинаковые боковые поля
- одинаковая логика ширины секций
- одинаковое поведение карточек относительно границ экрана

## 5.3 Вертикальный rhythm

Нужно соблюдать различие между:
- отступом между секциями
- отступом между элементами внутри секции
- отступом внутри карточки

Принцип:
- между секциями воздуха больше
- внутри секции воздух плотнее
- внутри карточек отступы стабильны

## 5.4 Карточки

Карточка — базовая единица интерфейса Release 1.

Карточка должна иметь:
- белый фон
- мягкий радиус
- легкую границу или очень мягкую тень
- стабильный внутренний padding
- четкий смысл

Карточка не должна:
- быть слишком декоративной
- иметь сложную вложенность без необходимости
- использовать цветную заливку как основной способ выделения

## 5.5 Максимальная ширина контента

### Mobile
Один основной контейнер с безопасными боковыми полями.

### Desktop
Центральный content container комфортной ширины.

Desktop не должен:
- растягивать контент на всю ширину окна без ограничений
- превращать формы в очень длинные строки
- ломать card-based логику mobile

## 6. Shell и навигация

## 6.1 Mobile shell

Mobile shell для Release 1:
- верхний header
- основная контентная область
- нижняя tab bar

### Header
На mobile header должен быть простым:
- либо page title
- либо back arrow + title
- при необходимости один action справа

Нельзя:
- перегружать header иконками
- ставить несколько равнозначных действий
- делать сложный header с несколькими строками без необходимости

### Bottom navigation
Нижняя навигация — основной способ перемещения между root-разделами.

Базовые разделы:
- `Главная`
- `Матчи`
- `Турниры`
- `Рейтинг`
- `Профиль`

Правила:
- активный таб выделяется синим
- остальные остаются нейтральными
- detail flows не должны создавать новые глобальные нижние табы

## 6.2 Desktop shell

Desktop shell для Release 1:
- верхняя горизонтальная навигация
- контент под top bar
- без левой глобальной sidebar на Release 1

Структура:
- слева бренд / название
- рядом или по центру основные разделы
- справа один главный action или важный контекстный action

Desktop shell не должен:
- становиться тяжелой корпоративной админкой
- добавлять второй глобальный навигационный уровень
- конфликтовать по логике с mobile shell

## 6.3 Правило адаптации

Desktop — это:
- более просторная адаптация mobile-shell
- а не новый отдельный продукт

Это означает:
- те же смысловые блоки
- те же сценарии
- те же состояния
- тот же визуальный язык

---

## 7. Навигационные правила

## 7.1 Root navigation

Root navigation существует только для основных разделов продукта.

Нельзя:
- плодить новые глобальные разделы без продуктовой необходимости
- превращать локальные фильтры и табы в глобальную навигацию

## 7.2 Detail flow

Detail flow идет через:
- back arrow в header
- или breadcrumb-like логику на desktop, если она нужна позже

Detail flow не должен:
- добавлять новый глобальный уровень навигации
- ломать нижнюю навигацию на mobile

## 7.3 Local tabs и filters

Локальные табы и фильтры допустимы:
- внутри страницы
- внутри секции
- если это реально помогает сценарию

Примеры:
- `Мои / Все` в турнирах

Но они должны:
- быть визуально вторичны по отношению к глобальной навигации
- иметь единый стиль во всех разделах

## 8. Основные UI-компоненты

## 8.1 Buttons

### Primary button
Используется для:
- главного действия экрана
- главного подтверждения внутри сценария

Стиль:
- синий фон
- белый текст
- четкий, но спокойный контраст

### Secondary button
Используется для:
- вторичных действий
- безопасных альтернатив
- локальных управляющих действий

Стиль:
- светлая или белая поверхность
- граница
- нейтральный текст

### Ghost / text action
Используется для:
- ссылочных действий
- `Отправить код повторно`
- `Отмена`
- `Полный список`

Правила:
- на одном экране не должно быть нескольких визуально равных primary actions без необходимости
- destructive action не должен выглядеть как обычный primary
- disabled state должен быть понятным, но мягким

## 8.2 Inputs

Базовый стиль input:
- светлая поверхность
- тонкая граница
- мягкий радиус
- label сверху
- понятное error-state поведение

Обязательные принципы:
- одинаковая высота полей в продукте
- одинаковое focus state поведение
- одинаковая логика ошибок
- одинаковая визуальная плотность

## 8.3 Status badges

Статус должен отображаться как компактный бейдж.

Правила:
- короткий текст
- компактный размер
- мягкий радиус
- без избыточной декоративности

Бейдж не должен:
- выглядеть как главная карточка
- занимать слишком много внимания
- быть просто декоративной плашкой

## 8.4 Cards

Основные типы карточек:
- auth-card
- tournament-card
- info-card
- match-card
- compact list card

Правило:
карточки могут отличаться содержанием, но должны ощущаться частью одной системы.

## 8.5 Form sections

Формы должны строиться как:
- понятные вертикальные секции
- с ровной логикой label → control → helper/error
- без визуальной перегрузки

## 9. Состояния интерфейса

## 9.1 Empty state

Empty state должен быть:
- текстовым
- спокойным
- коротким
- полезным

Допустимо:
- заголовок
- короткая поясняющая строка
- action, если он реально доступен роли

Недопустимо:
- иллюстрации
- эмодзи
- “маркетинговый” тон
- чрезмерный драматизм

## 9.2 Error state

Ошибка должна выглядеть как рабочее состояние, а не катастрофа.

Правила:
- локальная ошибка показывается рядом с полем или внутри блока
- общий экран ошибки должен быть редким
- ошибка не должна занимать больше внимания, чем нужно

## 9.3 Loading state

Предпочтительно:
- skeleton
- placeholder layout
- сохранение будущей структуры экрана

Желательно избегать:
- полноэкранных спиннеров без контекста
- скачков layout при загрузке

## 9.4 Disabled state

Disabled state должен:
- быть визуально понятным
- не выглядеть сломанным
- не спорить с активными элементами

---

## 10. Правила для page actions

## 10.1 Mobile

На mobile главный action должен быть:
- либо в header
- либо в нижнем sticky action area
- либо внутри основной карточки действия

Но не нужно одновременно дублировать его в нескольких местах без причины.

## 10.2 Desktop

На desktop главный action обычно находится:
- справа в верхней части страницы
- или в верхней части основной секции

## 10.3 One primary action rule

На одном экране или в одной секции должен быть один главный action.

Исключения допустимы только если это действительно два независимых сценария, что для Release 1 должно быть редкостью.

---

## 11. Правила для модулей

## 11.1 Auth

Auth — это:
- centered layout
- одна card
- один сценарий на экран
- минимум отвлекающих деталей

Auth не должен:
- задавать отдельный стиль продукта
- становиться “спец-экраном” с уникальным дизайном

## 11.2 Tournaments

Турниры задают базовый паттерн object-detail screen:
- заголовок
- статус
- блок параметров
- смысловые секции ниже

Это уже утвержденный pattern и его нужно использовать дальше.

## 11.3 Profile

Профиль должен выглядеть как:
- спокойная персональная страница
- без соцсети
- без больших аватаров
- без “identity-first” подачи

## 11.4 Ranking

Рейтинг должен делать акцент на:
- читаемости строк
- именах
- Elo
- порядке

Рейтинг не должен:
- использовать медали
- использовать шоу-элементы
- выглядеть как геймифицированный leaderboard

## 11.5 Matches

Матчи должны быть:
- компактными
- легко сканируемыми
- ориентированными на счет, участников и дату

## 11.6 Home

Главная — это обзор, а не dashboard.

На главной:
- только несколько ключевых блоков
- каждый блок должен вести в свой полноценный раздел
- не нужно пытаться уместить весь продукт на один экран

---

## 12. Desktop adaptation rules

## 12.1 Общий принцип

Desktop адаптация не должна менять продуктовую модель.

Она должна:
- давать больше воздуха
- улучшать читаемость
- позволять использовать 2 колонки там, где это полезно

## 12.2 Что можно делать на desktop

Допустимо:
- раскладывать info blocks в 2 колонки
- ставить соседние секции рядом, если они логически связаны
- делать списки шире
- показывать строки в более свободном формате

## 12.3 Что нельзя делать на desktop

Нельзя:
- превращать любой экран в dashboard
- добавлять sidebars без продуктовой причины
- делать desktop-only действия
- создавать radically different UX относительно mobile

---

## 13. Что уже считается зафиксированным

Следующие решения уже считаются принятыми и не должны переизобретаться без явной причины:

- светлый холодно-нейтральный фон
- белые карточки как основная surface model
- один основной синий акцент
- мягкие радиусы
- спокойная SaaS-типографика без декоративности
- mobile-first подход
- нижняя навигация на mobile
- верхняя навигация на desktop
- card-based composition
- локальные статусы через бейджи
- один главный CTA на экран или смысловой блок
- отсутствие графиков, иллюстраций, больших аватаров и спортивного пафоса
- auth и tournaments уже задают визуальный язык для остальных модулей
- пустые состояния должны быть простыми, текстовыми и функциональными

---

## 14. Что еще нужно фиксировать по мере развития

Следующие вещи могут быть дополнительно уточнены позже, но уже в рамках этого визуального подхода:

- точная типографическая шкала
- точные размеры spacing scale
- точные значения радиусов
- hover / focus / active / disabled states
- локальные паттерны таблиц и list rows
- точный стиль filters / local tabs
- точные loading skeletons
- desktop rules для сложных экранов

---

## 15. Правило для Codex и дальнейшей реализации

При работе над UI Codex должен:
- опираться на `UI_FOUNDATION.md`
- не изобретать новый стиль заново
- не добавлять consumer/sports/dashbord-like элементы
- не расширять scope продукта через дизайн
- использовать уже утвержденные паттерны auth + tournaments как базу для следующих экранов

Если по экрану нет точного макета:
- использовать принципы из этого документа
- выбирать самый спокойный и минималистичный вариант
- не добавлять декоративные решения по собственной инициативе
