# PRD — SideQuest Release 1

## 1. Обзор продукта

**SideQuest** — внутреннее веб-приложение для сотрудников компании, предназначенное для организации и учета офисных активностей и соревнований.

В **Release 1** пользовательский интерфейс и пользовательские сценарии ориентированы только на **настольный теннис**.

При этом архитектура и модель данных должны быть спроектированы так, чтобы в будущем платформу можно было расширить на другие активности, например:

- бильярд
- дартс
- кикер
- онлайн-игры
- другие внутренние активности

---

## 2. Цель Release 1

Сделать **внутренний MVP**, который позволит сотрудникам:

- войти в систему по email с одноразовым кодом подтверждения
- автоматически получить профиль при первом входе
- видеть свой профиль
- просматривать рейтинг игроков
- записывать результаты обычных матчей
- просматривать список турниров и страницы турниров
- организатору создавать и вести турниры
- видеть турнирную сетку
- администратору управлять пользователями и ролями

---

## 3. Основные продуктовые принципы Release 1

Release 1 должен быть:

- небольшим по объему
- понятным по логике
- простым в поддержке
- стабильным по поведению
- пригодным для последовательного расширения

Обязательные принципы Release 1:

- пользовательский интерфейс Release 1 работает только для настольного тенниса
- модель данных должна быть пригодна для будущего расширения на другие активности
- в Release 1 нельзя добавлять функции, которые не нужны для первого релиза
- Release 1 не должен превращаться в большую универсальную платформу до появления реальной потребности

---

## 4. Язык интерфейса и требования к адаптивности

### 4.1 Язык интерфейса

**Весь пользовательский интерфейс приложения должен быть на русском языке.**

Это включает:

- названия страниц
- кнопки
- подписи полей
- формы
- таблицы
- статусы
- сообщения об ошибках
- пустые состояния
- тексты подтверждения действий
- навигацию
- служебные тексты на экранах

Допустимо:

- технические названия в коде — на английском языке
- названия таблиц, полей базы данных и внутренних сущностей — на английском языке

Недопустимо:

- смешанный русский и английский язык в видимом пользовательском интерфейсе
- английские placeholder’ы, кнопки или статусы в production UI Release 1

### 4.2 Адаптивность интерфейса

Веб-приложение должно быть адаптивным и удобным для использования на мобильных устройствах.

Для Release 1 мобильный сценарий использования является приоритетным, так как значительная часть пользовательских действий будет происходить с телефона.

Интерфейс должен быть удобен на мобильных устройствах для следующих действий:

- вход в систему
- просмотр главной страницы
- просмотр рейтинга
- просмотр списка турниров
- просмотр страницы турнира
- создание обычного матча
- ввод результата матча
- просмотр собственного профиля

Обязательные требования Release 1:

- основные действия должны быть доступны без desktop-only взаимодействий
- кнопки и интерактивные элементы должны быть удобны для нажатия с телефона
- формы не должны быть перегружены
- страницы не должны ломаться на небольших экранах
- критические пользовательские сценарии должны быть удобны как на desktop, так и на мобильных устройствах

## 5. Модель доступа и роли Release 1

### 5.1 Доступ к приложению

В Release 1 без авторизации доступна только:

- страница входа

Все страницы приложения, кроме страницы входа, требуют авторизации.

Вход в систему выполняется по `email + одноразовый код подтверждения`.

В Release 1 пользователь может войти в систему с любым email, если успешно подтверждает вход одноразовым кодом и его доступ не был отключен администратором.

### 5.2 Роли

В Release 1 у каждого пользователя есть **ровно одна системная роль**.

Поддерживаются три роли:

- `player`
- `organizer`
- `admin`

Пользователь не выбирает роль сам и не переключает роли внутри системы.

### 5.3 Player

Обычный участник.

Может:

- войти в систему
- видеть главную страницу
- видеть свой профиль
- видеть рейтинг
- видеть список турниров
- видеть страницу турнира
- создавать обычный матч только со своим участием
- записывать результат обычного матча только со своим участием

Не может:

- создавать турниры
- управлять турнирами
- управлять пользователями
- менять роли

### 5.4 Organizer

Организатор.

Может все, что `player`, а также:

- создавать турниры
- управлять только собственными турнирами
- добавлять участников в собственный турнир
- удалять участников из собственного турнира до его старта
- запускать генерацию сетки собственного турнира
- вносить результаты матчей собственного турнира
- завершать собственный турнир
- отменять собственный турнир до завершения

Не может:

- управлять чужими турнирами
- менять роли пользователей
- управлять пользователями системы

### 5.5 Admin

Администратор.

Может все, что `organizer`, а также:

- просматривать список пользователей
- менять роль пользователя
- активировать и деактивировать пользователя
- просматривать базовую административную информацию о пользователях
- управлять любыми турнирами независимо от автора создания

В Release 1 `admin` не обязан иметь отдельные UI-инструменты для ручного редактирования матчей, рейтинга или исторических данных.

### 5.6 Деактивация пользователей

Деактивированный пользователь не может использовать систему.

Это означает:

- он не может войти в систему
- он не может создавать матчи
- он не может создавать турниры
- он не может управлять турнирами
- он не может выполнять административные действия

Исторические данные деактивированного пользователя сохраняются.

### 5.7 Защита последнего active admin

В Release 1 система не должна позволять:

- понизить роль последнего активного пользователя с ролью `admin`
- деактивировать последнего активного пользователя с ролью `admin`

## 6. Аутентификация и создание пользователя

### 6.1 Способ входа

В Release 1 используется только один способ входа:

- ввод email
- отправка одноразового кода подтверждения на почту
- ввод полученного кода в течение срока его действия
- создание или восстановление сессии после успешного подтверждения

### 6.1.1 Правила одноразового кода подтверждения

В Release 1 одноразовый код подтверждения является обязательной частью входа по email.

Обязательные правила:

- код подтверждения является одноразовым
- код подтверждения действует **10 минут** с момента отправки
- после истечения срока действия код считается недействительным
- после успешного входа использованный код повторно использовать нельзя
- пользователь может запросить новый код подтверждения
- повторная отправка кода не должна быть доступна чаще одного раза в **30 секунд**
- для одного кода должно быть доступно не более **5 попыток ввода**
- после превышения допустимого количества попыток текущий код считается недействительным, и пользователь должен запросить новый

### 6.2 Что не используется в Release 1

В Release 1 не используются:

- пароль
- magic link как основной способ входа
- социальные логины
- MFA
- ручная регистрация через отдельную форму
- самостоятельный выбор роли пользователем

### 6.3 Email для входа

В Release 1 пользователь может войти в систему с любым корректным email.

Это означает:

- email не ограничивается корпоративными доменами
- email может быть рабочим или личным
- домен email сам по себе не ограничивает доступ к приложению

Доступ в систему определяется успешным подтверждением входа одноразовым кодом и текущим статусом пользователя.

Если пользователь деактивирован администратором, система не должна пускать его в приложение даже при корректном вводе email и кода.

### 6.4 Первый вход нового пользователя

При первом успешном входе нового пользователя система должна:

- создать профиль пользователя
- сохранить email пользователя
- назначить роль `player`
- пометить пользователя как активного

### 6.5 Назначение ролей

Правила назначения ролей в Release 1:

- базовая роль нового пользователя — `player`
- роль `player` назначается автоматически
- роли `organizer` и `admin` назначаются только администратором
- пользователь не может сам повысить себе роль
- в системе нет режима множественных ролей на одного пользователя

## 7. Функциональный объем Release 1

### 7.1 Главная страница

Главная страница должна показывать:

- название платформы
- краткое описание продукта
- блок с рейтингом или топом игроков
- блок с последними матчами
- блок с актуальными турнирами
- основные действия, доступные текущей роли

Главная страница не должна превращаться в сложный аналитический дашборд.

### 7.2 Профиль пользователя

Профиль пользователя должен содержать:

- имя
- email
- роль
- текущий рейтинг
- количество сыгранных матчей
- последние матчи
- участие в турнирах

В Release 1 профиль не включает:

- аватар
- уведомления
- расширенные настройки
- достижения
- загрузку медиа
- сложную персональную аналитику

### 7.3 Рейтинг

Система должна:

- хранить рейтинг участника по активности
- отображать рейтинг на отдельной странице
- сортировать рейтинг по убыванию значения
- показывать как минимум:
  - место
  - имя
  - рейтинг
  - количество матчей

#### Правила рейтинга Release 1

В Release 1 используется только **система Elo**.

Фиксированные правила Elo для Release 1:

- стартовый рейтинг нового участника: `1000`
- `K-factor`: `32`
- рейтинг пересчитывается только после завершенного матча
- один завершенный матч дает одно обновление рейтинга
- обычные матчи и турнирные матчи используют один и тот же Elo-алгоритм
- отмененные матчи не влияют на рейтинг
- `bye` в турнирной сетке не влияет на рейтинг

В Release 1 не используются:

- сезонные рейтинги
- отдельные турнирные рейтинги
- альтернативные рейтинговые системы
- дополнительные коэффициенты сложности

### 7.4 Формат матча Release 1

В Release 1 формат матча выбирается из фиксированного набора:

- `BO1`
- `BO3`
- `BO5`

Это правило распространяется на:

- обычные матчи
- турнирные матчи

#### Правила результата матча

В Release 1 в матче хранится итоговый счет **по партиям**, а не по очкам внутри партии.

Допустимые итоговые результаты зависят от выбранного формата:

- для `BO1` допустим только результат `1:0` или `0:1`
- для `BO3` допустим только результат `2:0`, `2:1`, `0:2` или `1:2`
- для `BO5` допустим только результат `3:0`, `3:1`, `3:2`, `0:3`, `1:3` или `2:3`

Фиксированные правила Release 1:

- ничья не поддерживается
- каждый завершенный матч должен иметь ровно одного победителя
- итоговый счет должен соответствовать выбранному формату матча
- победитель должен соответствовать итоговому счету
- матч нельзя сохранить без выбранного формата
- матч нельзя сохранить с результатом, который не соответствует формату

### 7.5 Обычные матчи

Обычные матчи — это матчи вне турнирной сетки.

#### Правила обычных матчей Release 1

Обычный матч в Release 1:

- всегда матч `1 на 1`
- создается сразу вместе с финальным результатом
- не проходит отдельную стадию планирования
- не требует подтверждения вторым участником
- после сохранения сразу влияет на рейтинг

#### Что должно храниться у обычного матча

Система должна хранить:

- активность
- первого участника
- второго участника
- формат матча
- итоговый счет по партиям
- победителя
- дату создания записи
- пользователя, который создал запись

#### Ограничения обычных матчей Release 1

- игрок может создать обычный матч только со своим участием
- игрок не может создать матч между двумя другими пользователями
- матч с самим собой запрещен
- матч нельзя сохранить без победителя
- матч нельзя сохранить без итогового результата
- матч нельзя редактировать через UI после сохранения
- матч нельзя удалять через UI после сохранения

#### Что не входит в Release 1 для обычных матчей

- планирование матчей
- статус "идет сейчас"
- подтверждение результата вторым игроком
- спорные результаты
- комментарии к матчу
- вложения
- покадровый или поочковый протокол партий

### 7.6 Турниры

Турнир — это отдельное соревнование в рамках одной активности.

В Release 1 пользовательский интерфейс турниров поддерживает только настольный теннис.

#### Поддерживаемый формат турнира

В Release 1 поддерживается только один формат турнира:

- `single elimination`

Другие форматы турниров в Release 1 не поддерживаются.

#### Формат турнирных матчей

При создании турнира организатор должен выбрать единый формат турнирных матчей:

- `BO1`
- `BO3`
- `BO5`

Выбранный формат применяется ко всем матчам данного турнира.

После старта турнира формат матчей турнира менять нельзя.

#### Статусы турнира Release 1

Турнир может иметь только следующие статусы:

- `draft`
- `in_progress`
- `completed`
- `cancelled`

#### Доступ к турнирам

Все авторизованные пользователи могут:

- просматривать список турниров
- открывать страницу турнира в режиме просмотра

Права на управление турниром определяются ролью пользователя и владением турниром.

#### Правило владения турниром

В Release 1 `organizer` может управлять только теми турнирами, которые создал сам.

Это означает, что `organizer` может только для собственных турниров:

- редактировать турнир
- добавлять участников
- удалять участников до старта
- генерировать сетку
- вносить результаты турнирных матчей
- завершать турнир
- отменять турнир

`organizer` не должен иметь доступа к управлению чужими турнирами.

`admin` может просматривать и управлять любыми турнирами независимо от автора создания.

#### Создание турнира

Организатор должен иметь возможность:

- создать турнир
- указать название турнира
- указать активность
- выбрать формат турнирных матчей
- при необходимости указать дату, время и локацию
- сохранить турнир в статусе `draft`

Добавление участников выполняется после создания турнира на странице турнира в статусе `draft`.

#### Участники турнира

Правила участников турнира в Release 1:

- участники турнира добавляются организатором вручную
- самостоятельная регистрация пользователей в турнир не поддерживается
- удаление участника возможно только до старта турнира
- после начала турнира состав участников нельзя менять

#### Генерация сетки

После подготовки турнира организатор должен иметь возможность:

- запустить генерацию сетки
- перевести турнир в статус `in_progress`

Правила генерации сетки:

- сетка генерируется автоматически
- участники распределяются автоматически
- при нехватке участников для ровной сетки допускается использование `bye`
- `bye` не влияет на рейтинг
- после генерации сетки турнир считается начатым

#### Турнирные матчи

Система должна:

- хранить матчи внутри турнира отдельно от обычных матчей
- хранить участников турнирного матча
- хранить формат турнирного матча
- хранить результат турнирного матча
- хранить победителя турнирного матча
- продвигать победителя по сетке после сохранения результата

#### Правила результата турнирного матча Release 1

- в Release 1 ничья в турнирных матчах не поддерживается
- каждый завершенный турнирный матч должен иметь ровно одного победителя
- итоговый счет турнирного матча должен соответствовать формату турнира

#### Кто может вносить результаты турнирных матчей

В Release 1 результаты турнирных матчей могут вносить только:

- `organizer`
- `admin`

#### Завершение турнира

Турнир может быть завершен только после определения финального победителя.

После завершения:

- турнир получает статус `completed`
- редактирование состава участников через UI прекращается
- редактирование структуры сетки через UI прекращается

#### Что не входит в Release 1 для турниров

- self-registration в турнир
- round robin
- double elimination
- командные турниры
- mixed-форматы
- ручная перестройка сетки после старта
- сложные правила посева
- несколько активностей в UI
- спорные результаты турнирных матчей

### 7.7 Администрирование пользователей

В Release 1 административный раздел включает только управление пользователями.

Администратор должен иметь возможность:

- видеть список пользователей
- видеть email пользователя
- видеть текущую роль пользователя
- видеть, активен ли пользователь
- менять роль пользователя
- активировать пользователя
- деактивировать пользователя

В Release 1 административный раздел не включает:

- редактирование матчей

## 8. Поддержка активностей в Release 1

Release 1 должен использовать **универсальную модель данных для активностей**, но UI и пользовательские сценарии должны быть ограничены одной активностью:

- настольный теннис

Обязательные правила:

- в базе данных должна существовать сущность типа активности
- в seed data Release 1 должна быть создана как минимум одна активность — настольный теннис
- пользовательский интерфейс Release 1 не должен пытаться полноценно поддерживать несколько разных активностей
в Release 1 не требуется отдельная административная страница для управления активностями

## Контекст текущей активности Release 1

В **Release 1** пользовательский интерфейс работает только для одной активности — **настольного тенниса**.

При этом все основные пользовательские данные должны трактоваться не как "общие по системе", а как данные **в контексте текущей активности**.

Это означает, что в Release 1:

- рейтинг отображается **по текущей активности**
- последние матчи отображаются **по текущей активности**
- список обычных матчей отображается **по текущей активности**
- турниры отображаются **по текущей активности**
- профиль пользователя показывает статистику и участие **по текущей активности**, если не указано иное

В Release 1 текущая активность фиксирована и равна:

- `table_tennis`
- **Настольный теннис**

### Правило отображения данных Release 1

Даже если в пользовательском интерфейсе Release 1 не показывается отдельный фильтр или переключатель активности, система все равно должна считать, что все основные списки и агрегаты на пользовательских экранах строятся **в рамках текущей активности**, а не по всем активностям сразу.

Недопустимо для Release 1:

- смешивать в одном рейтинге данные разных активностей
- смешивать в одном списке матчей данные разных активностей
- смешивать в одном списке турниров данные разных активностей
- трактовать главные пользовательские экраны как глобальные экраны "по всей платформе" без activity scope

### Ограничение интерфейса Release 1

В Release 1 отдельный UI-фильтр или переключатель активности **не требуется**, так как пользовательский интерфейс работает только для настольного тенниса.

Однако это не отменяет обязательного правила: основные данные на экранах должны быть технически привязаны к текущей активности.

### Требование к будущему расширению

Архитектура Release 1 должна позволять в будущем добавить выбор активности без переписывания базовой логики экранов.

Для этого уже в Release 1 необходимо считать, что:

- у экранов есть контекст текущей активности
- рейтинг, матчи и турниры выбираются по activity scope
- расширение на новые активности должно происходить через смену текущей активности, а не через переписывание существующих экранов с нуля

## 13. Фиксированные бизнес-правила Release 1

Ниже перечислены правила, которые должны считаться обязательными и не должны трактоваться свободно:

1. В Release 1 у пользователя только одна роль.
2. Новый пользователь автоматически получает роль `player`.
3. Роли `organizer` и `admin` назначаются только администратором.
4. Вход в систему выполняется по email и одноразовому коду подтверждения.
5. Вход в Release 1 возможен с любым корректным email через одноразовый код подтверждения.
6. Все страницы приложения, кроме страницы входа, требуют авторизации.
7. Весь пользовательский интерфейс должен быть на русском языке.
8. В Release 1 UI работает только для настольного тенниса.
9. Обычный матч — это матч `1 на 1`, который создается сразу с финальным результатом.
10. Игрок может создать обычный матч только со своим участием.
11. Матч с самим собой запрещен.
12. В Release 1 используется только система Elo.
13. Стартовый рейтинг участника — `1000`.
14. `K-factor` для Elo — `32`.
15. Турнир в Release 1 поддерживает только `single elimination`.
16. Участники турнира добавляются только организатором.
17. `organizer` может управлять только собственными турнирами.
18. `admin` может управлять любым турниром.
19. Любой авторизованный пользователь может просматривать список турниров и страницу турнира.
20. Результаты турнирных матчей вносят только `organizer` или `admin`.
21. `bye` не влияет на Elo.
22. Завершенные матчи не редактируются и не удаляются через UI.
23. В Release 1 нет отдельной UI-страницы управления активностями.
24. Деактивированный пользователь не может войти в систему и выполнять новые действия.
25. Последний активный `admin` не может быть понижен или деактивирован.
26. В Release 1 ничья не поддерживается; каждый завершенный матч должен иметь ровно одного победителя.
27. Формат матча выбирается только из `BO1`, `BO3`, `BO5`.
28. Итоговый счет должен соответствовать выбранному формату матча.
29. Release 1 не должен содержать функциональность, явно перечисленную как out of scope.
30. Одноразовый код подтверждения действует 10 минут, является одноразовым и не может использоваться повторно после успешного входа.
31. Повторная отправка кода подтверждения не должна быть доступна чаще одного раза в 30 секунд.
32. Для одного кода допускается не более 5 попыток ввода.

## 14. Критерии готовности Release 1

Release 1 считается готовым, если выполняются все условия ниже:

1. Пользователь может войти в систему по email с одноразовым кодом подтверждения.
2. Пользователь может войти в систему с любым корректным email через одноразовый код подтверждения.
3. При первом входе автоматически создается профиль пользователя.
4. Новому пользователю автоматически назначается роль `player`.
5. Пользователь может открыть свой профиль после входа.
6. Пользователь может открыть страницу рейтинга.
7. Рейтинг отображается на отдельной странице.
8. Рейтинг рассчитывается по системе Elo.
9. Стартовый рейтинг нового участника равен `1000`.
10. `K-factor` равен `32`.
11. Пользователь роли `player` может создать обычный матч только со своим участием.
12. Система не позволяет создать матч с самим собой.
13. Пользователь может выбрать формат обычного матча из `BO1`, `BO3`, `BO5`.
14. После сохранения обычного матча Elo обновляется.
15. Организатор может создать турнир.
16. Организатор может выбрать формат турнира из `BO1`, `BO3`, `BO5`.
17. Организатор может добавить участников турнира.
18. Организатор может сгенерировать сетку `single elimination`.
19. Организатор или администратор может внести результат турнирного матча.
20. Победитель турнирного матча автоматически продвигается по сетке.
21. Турнир можно завершить после определения победителя.
22. Организатор может управлять только собственными турнирами.
23. Администратор может управлять любым турниром.
24. Любой авторизованный пользователь может просматривать список турниров и страницу турнира.
25. Администратор может видеть список пользователей.
26. Администратор может менять роль пользователя.
27. Администратор может активировать и деактивировать пользователя.
28. Система не позволяет понизить или деактивировать последнего активного администратора.
29. Деактивированный пользователь не может войти в систему.
30. Все страницы приложения, кроме страницы входа, требуют авторизации.
31. Весь пользовательский интерфейс отображается на русском языке.
32. Система не позволяет сохранить результат, который не соответствует выбранному формату матча.
33. В приложении отсутствуют функции, явно перечисленные как out of scope для Release 1
34. Одноразовый код подтверждения имеет TTL 10 минут.
35. Просроченный код не позволяет войти в систему.
36. Уже использованный код не позволяет войти в систему повторно.
37. Повторная отправка кода ограничена интервалом 30 секунд.
38. После превышения 5 попыток ввода текущий код становится недействительным.
