# TECH_STACK.md — SideQuest Release 1

## 1. Назначение документа

Этот документ фиксирует **технический стек, инфраструктурные решения и технические ограничения** проекта **SideQuest Release 1**.

Он определяет:

- какие технологии используются в проекте
- какие технологии нельзя менять без явного запроса
- как устроена локальная среда разработки
- какие технические решения считаются обязательными
- какие библиотеки и подходы допустимы, а какие — нет

Этот документ **не** описывает:

- продуктовые требования
- доменную модель
- роли и права
- порядок этапов реализации
- правила поведения Codex

Для этого используются:

- `PRD.md`
- `DOMAIN_MODEL.md`
- `RBAC.md`
- `TASKS.md`
- `AGENTS.md`

---

## 2. Общие принципы технического стека

Release 1 должен использовать **простой, понятный и устойчивый стек**, достаточный для MVP.

Главные принципы:

- использовать минимально достаточный набор технологий
- не усложнять стек без реальной необходимости
- не менять базовые технические решения без явного запроса
- не добавлять тяжелые инструменты “на будущее”
- не строить инфраструктуру, рассчитанную на значительно больший масштаб, чем нужен Release 1

---

## 3. Базовый стек проекта

В Release 1 используется следующий стек.

### Приложение
- **Next.js**
- **React**
- **TypeScript**

### UI
- **Tailwind CSS**

### База данных
- **PostgreSQL**

### Локальная инфраструктура
- **Docker** для локального запуска базы данных

### Доступ к данным и миграции
- **Drizzle ORM**
- **Drizzle Kit**
- **SQL migrations через Drizzle**

### Runtime и package manager
- **Node.js**
- **npm**

---

## 4. Фиксированные технические решения Release 1

Следующие решения считаются обязательными и не должны свободно заменяться:

1. Приложение реализуется на **Next.js**
2. Код приложения пишется на **TypeScript**
3. Основная стилизация строится на **Tailwind CSS**
4. Основная БД проекта — **PostgreSQL**
5. Локальная БД запускается через **Docker**
6. Работа со схемой БД и доступом к данным строится через **Drizzle**
7. Изменения схемы выполняются через **миграции**
8. Локальная конфигурация проекта использует `.env.local`
9. Текущая рабочая инфраструктура проекта не должна быть разрушена

---

## 5. Технические слои приложения

### 5.1 UI layer

UI-слой строится внутри Next.js-приложения.

Назначение слоя:

- страницы
- layout
- навигация
- формы
- отображение данных
- role-aware UI
- русскоязычные тексты интерфейса

Ограничения:

- UI не должен содержать тяжелую бизнес-логику
- UI не должен быть источником истины по правам доступа
- UI не должен реализовывать продуктовые правила вместо серверной логики

### 5.2 Server layer

Серверная логика реализуется внутри Next.js-приложения.

Назначение слоя:

- чтение данных
- изменение данных
- серверные проверки доступа
- простая бизнес-логика Release 1
- работа с БД через Drizzle

Ограничения:

- не строить отдельный backend-сервис вне Next.js без явного запроса
- не вводить микросервисную архитектуру
- не выносить бизнес-логику в отдельную распределенную инфраструктуру

### 5.3 Database layer

Слой данных строится на PostgreSQL + Drizzle.

Назначение слоя:

- таблицы Release 1
- связи
- ограничения
- миграции
- seed data
- чтение и запись данных

Ограничения:

- не использовать другую основную БД
- не хранить доменную модель в local JSON / file storage вместо PostgreSQL
- не обходить миграции при изменении схемы

## 6. Требования к UI-реализации

Техническая реализация UI должна учитывать, что в `PRD.md` мобильный сценарий использования является приоритетным.

Следствия для реализации:

- интерфейс должен быть адаптивным
- экраны нельзя проектировать только под desktop
- формы, таблицы и основные действия должны быть пригодны для мобильного использования
- при реализации новых экранов нужно учитывать mobile-first или mobile-priority подход
- интерактивные элементы должны оставаться usable на небольших экранах

Этот раздел не заменяет продуктовые требования из `PRD.md`, а задает технический способ их реализации.

## 7. Next.js

### 7.1 Назначение

Next.js является основой приложения Release 1.

Используется для:

- маршрутов приложения
- серверного рендеринга
- серверных компонентов и/или серверных действий
- UI
- страниц
- базовой серверной логики MVP

### 7.2 Обязательные правила

- использовать текущую структуру Next.js-проекта
- не заменять Next.js на другой framework
- не выносить приложение в separate frontend + separate backend без явного запроса
- не строить отдельный API backend “на всякий случай”

### 7.3 Что нельзя делать

Нельзя без запроса:

- мигрировать проект на другой framework
- вводить отдельную BFF-архитектуру
- строить отдельный Node/Express backend параллельно Next.js
- делить проект на несколько приложений

---

## 8. TypeScript

### 8.1 Назначение

TypeScript обязателен для всего приложения.

### 8.2 Обязательные правила

- использовать TypeScript во всех новых модулях
- использовать явные и понятные типы
- не писать новый production-код на JavaScript, если это не техническая необходимость
- не отключать type safety без явной причины

### 8.3 Предпочтения

Предпочтительно:

- небольшие и понятные типы
- локально понятные интерфейсы
- композиция типов вместо чрезмерной generic-сложности

Нежелательно:

- чрезмерно сложные generics без пользы
- типизация “на вырост”, которую никто не использует
- типы, скрывающие доменную структуру вместо прояснения

---

## 9. Tailwind CSS

### 9.1 Назначение

Tailwind CSS — базовый инструмент стилизации UI.

### 9.2 Обязательные правила

- использовать Tailwind как основной способ стилизации
- не вводить второй основной styling system без запроса
- не смешивать несколько полноценных систем стилизации без причины

### 9.3 Что допустимо

Допустимо:

- utility-based стили
- выделение переиспользуемых UI-компонентов
- аккуратные общие классы или helpers, если они реально уменьшают дублирование

### 9.4 Что нельзя делать

Нельзя без запроса:

- заменять Tailwind на другой основной подход
- вводить тяжелые CSS-in-JS решения
- строить отдельный дизайн-фреймворк внутри MVP

---

## 10. PostgreSQL

### 10.1 Назначение

PostgreSQL — основная база данных проекта.

### 10.2 Обязательные правила

- все основные данные Release 1 должны храниться в PostgreSQL
- доменная модель Release 1 не должна храниться в файловой системе или browser storage как основной источник истины
- схема базы должна поддерживаться миграциями

### 10.3 Что нельзя делать

Нельзя без запроса:

- менять PostgreSQL на другую БД
- дублировать доменную истину в local JSON files
- обходить БД ради “быстрого MVP”, если функциональность уже зависит от данных

---

## 11. Docker

### 11.1 Назначение

Docker используется для локального запуска PostgreSQL.

### 11.2 Текущее обязательное решение

Локальная база данных проекта должна запускаться через существующую docker-конфигурацию проекта.

### 11.3 Обязательные правила

- не ломать существующий `compose.yaml`
- не менять способ локального запуска БД без явного запроса
- не удалять существующие volumes, если это не требуется явно
- не затрагивать unrelated контейнеры и инфраструктуру

### 11.4 Что нельзя делать

Нельзя без запроса:

- переносить локальную БД с Docker на другой способ запуска
- перестраивать всю локальную инфраструктуру
- добавлять новые контейнеры без реальной необходимости текущей задачи

---

## 12. Drizzle ORM и Drizzle Kit

### 12.1 Назначение

Drizzle — основной способ:

- описания схемы БД
- доступа к данным
- построения запросов
- работы с миграциями

### 12.2 Обязательные правила

- использовать Drizzle как основной ORM / SQL access layer
- использовать Drizzle migrations для изменения схемы
- не заменять Drizzle на Prisma, TypeORM, Sequelize или другой ORM без явного запроса
- не писать новый слой данных в обход выбранного стека без необходимости

### 12.3 Что допустимо

Допустимо:

- использовать schema definitions через Drizzle
- использовать Drizzle queries
- использовать SQL там, где это оправдано и не ломает общую консистентность проекта

### 12.4 Что нельзя делать

Нельзя без запроса:

- строить второй полноценный ORM-слой параллельно Drizzle
- заводить repository framework “на будущее”
- смешивать несколько ORM в одном MVP

---

## 13. Миграции

### 13.1 Общий принцип

Все изменения схемы базы должны выполняться через миграции.

### 13.2 Обязательные правила

- не менять схему вручную в обход миграций
- не редактировать уже примененные миграции задним числом, если это можно избежать
- для новых изменений добавлять новые миграции
- проверять, что миграции применяются локально

### 13.3 Что нельзя делать

Нельзя без запроса:

- удалять старые миграции
- переписывать историю миграций ради “красоты”
- использовать ручные ad hoc SQL-изменения вместо миграций как основной путь

---

## 14. Переменные окружения

### 14.1 Общий принцип

Конфигурация проекта должна использовать переменные окружения.

### 14.2 Текущий обязательный механизм

Локальная разработка использует `.env.local`.

### 14.3 Обязательные правила

- использовать `.env.local` для локальных переменных окружения
- не хардкодить секреты, connection strings и чувствительные значения в коде
- не коммитить секреты в репозиторий
- использовать существующую переменную `DATABASE_URL` как стандартный способ подключения к БД

### 14.4 Что нельзя делать

Нельзя без запроса:

- переносить секреты в код
- использовать альтернативные локальные конфиги вместо существующего механизма
- ломать текущую рабочую конфигурацию окружения

---

## 15. Структура проекта

### 15.1 Общий принцип

Проект должен сохранять простую и читаемую структуру.

### 15.2 Обязательные правила

- не перестраивать структуру проекта без явного запроса
- не переносить модули массово “ради порядка”
- не раздувать количество папок и слоев
- не строить enterprise-структуру для MVP

### 15.3 Предпочтительный подход

Предпочтительно:

- логичное разделение UI, db, server logic
- ясные и небольшие модули
- локально понятная организация кода

Нежелательно:

- глубокая иерархия папок без пользы
- многослойная enterprise-структура
- дублирование кода из-за плохой структуры

- универсальные factory systems
- глобальный event bus
- plugin architecture
- EAV-модель для БД
- heavy abstraction around ORM
- чрезмерно generic utilities

### 17.2 Запрещено без явного запроса

Нельзя:

- менять основной стек
- менять БД
- менять ORM
- строить микросервисы
- выносить backend в отдельный сервис
- строить публичное API как отдельный большой слой

## 18. Добавление новых библиотек

### 18.1 Общий принцип

Новые библиотеки можно добавлять только если они действительно нужны для текущей задачи.

### 18.2 Обязательные правила

Перед добавлением новой библиотеки нужно убедиться, что:

- задача не решается средствами текущего стека
- библиотека не дублирует уже существующий инструмент
- библиотека не расширяет scope Release 1
- библиотека не создает лишнюю зависимость ради мелкой удобности

### 18.3 Что нельзя делать

Нельзя:

- добавлять библиотеку “на будущее”
- добавлять альтернативу уже выбранному инструменту
- подключать тяжелую зависимость ради одной маленькой функции, если есть простой вариант без нее

## 20. Локальная разработка

### 20.1 Общая модель

Локальная разработка строится так:

- приложение запускается локально
- PostgreSQL работает локально через Docker
- приложение подключается к БД через `DATABASE_URL`
- изменения схемы проводятся миграциями
- DBeaver или другой SQL GUI может использоваться только как вспомогательный инструмент просмотра, но не как основной механизм изменения схемы

### 20.2 Обязательные правила

- не считать ручные изменения через GUI основным способом изменения схемы
- использовать GUI только для просмотра и проверки, если нужно
- любые изменения схемы фиксировать через миграции
