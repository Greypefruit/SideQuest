# DOC_INDEX.md — SideQuest

## 1. Назначение документа

Этот файл является **главной навигационной картой документации проекта SideQuest**.

Перед началом любой задачи сначала нужно открыть этот файл, а затем читать остальные документы в указанном здесь порядке.

`DOC_INDEX.md` не заменяет содержательные документы проекта.  
Его задача — помочь быстро понять:

- какие документы существуют
- за что отвечает каждый документ
- в каком порядке их читать
- какой документ является источником истины для конкретного типа вопроса
- что делать, если документы кажутся пересекающимися

---

## 2. Общий принцип работы с документацией

В проекте используется разделение документов по типам ответственности.

Каждый документ отвечает только за свой слой:

- продукт
- доменная модель
- роли и доступ
- порядок реализации
- стек и инфраструктура
- валидации и edge cases
- дисциплина работы Codex

Нельзя пытаться искать все ответы в одном файле.  
Если вопрос относится к конкретному слою, нужно открывать документ, который отвечает именно за этот слой.

---

## 3. Обязательный порядок чтения документов

Перед выполнением любой задачи использовать документы в следующем порядке:

1. `PRD.md`
2. `DOMAIN_MODEL.md`
3. `RBAC.md`
4. `TASKS.md`
5. `TECH_STACK.md`
6. `VALIDATION_RULES.md`
7. `AGENTS.md`

Этот порядок обязателен, потому что:

- сначала нужно понять **что** строится
- затем **из каких сущностей** это состоит
- затем **кто и что может делать**
- затем **на каком этапе** проекта сейчас находится работа
- затем **на чем** это реализуется
- затем **какие проверки и негативные сценарии** обязательны
- и только после этого — **как именно Codex должен работать внутри проекта**

## 4. За что отвечает каждый документ

## 4.1 `PRD.md`

`PRD.md` — главный источник **продуктовых требований Release 1**.

Использовать `PRD.md`, если нужно понять:

- что должно делать приложение
- какие пользовательские сценарии входят в Release 1
- какие роли есть в продукте
- какие страницы и функции должны существовать
- какие бизнес-правила считаются обязательными
- что входит и что не входит в Release 1
- какие критерии готовности считаются обязательными

`PRD.md` отвечает на вопрос:

**Что строим и как должен вести себя продукт?**

---

## 4.2 `DOMAIN_MODEL.md`

`DOMAIN_MODEL.md` — главный источник по **сущностям, связям, таблицам, полям и доменным инвариантам**.

Использовать `DOMAIN_MODEL.md`, если нужно понять:

- какие сущности существуют в системе
- какие таблицы должны быть в базе данных
- какие поля обязательны
- какие связи допустимы
- какие состояния данных допустимы
- какие сущности не нужно добавлять в Release 1
- как устроены `profiles`, `participants`, `matches`, `competitions` и другие доменные объекты

`DOMAIN_MODEL.md` отвечает на вопрос:

**Из каких сущностей состоит система и какие данные считаются корректными?**

---

## 4.3 `RBAC.md`

`RBAC.md` — главный источник по **ролям и правам доступа**.

Использовать `RBAC.md`, если нужно понять:

- какие роли существуют в Release 1
- сколько ролей может быть у пользователя
- кто может создавать матчи
- кто может создавать турниры
- кто может управлять турнирами
- кто может управлять пользователями
- как устроено владение турнирами
- какие серверные проверки доступа обязательны
- что запрещено `player`, `organizer`, `admin`

`RBAC.md` отвечает на вопрос:

**Кто что может делать?**

---

`TASKS.md` — главный источник по **порядку реализации Release 1**.

Использовать `TASKS.md`, если нужно понять:

- какой этап проекта идет сейчас
- что должно быть сделано раньше, а что позже

- какие acceptance criteria у каждого этапа
- что входит и не входит в scope конкретной задачи реализации

`TASKS.md` отвечает на вопрос:

**В каком порядке строим проект и что именно делаем на текущем этапе?**

Использовать `TECH_STACK.md`, если нужно понять:

**На чем строим проект и какие технические ограничения уже зафиксированы?**

## 4.6 `VALIDATION_RULES.md`

`VALIDATION_RULES.md` — главный источник по **валидациям, негативным сценариям, edge cases, anti-duplicate и anti-race правилам**.

Использовать `VALIDATION_RULES.md`, если нужно понять:

- какие входные данные нужно валидировать
- какие пользовательские ошибки должны блокироваться

- как обрабатывать OTP-коды, TTL и лимиты попыток
- как защищать последнего active admin
- какие негативные сценарии обязательно должны быть закрыты

`VALIDATION_RULES.md` отвечает на вопрос:

**Какие невалидные сценарии должна блокировать система и как защищаться от поломок?**

## 5. Что делать, если вопрос относится сразу к нескольким слоям

Некоторые вопросы затрагивают сразу несколько документов.

В таком случае нужно разбирать вопрос по слоям, а не искать один “универсальный” ответ.

### Пример 1: “Можно ли `organizer` редактировать чужой турнир?”
Нужно смотреть:

- `PRD.md` — бизнес-правило
- `RBAC.md` — правило доступа
- `DOMAIN_MODEL.md` — источник владения через `created_by_profile_id`

### Пример 2: “Какие поля у матча обязательны?”
Нужно смотреть:

- `DOMAIN_MODEL.md` — структура сущности
- `PRD.md` — продуктовые правила поведения матча
- `VALIDATION_RULES.md` — какие значения запрещены

### Пример 3: “Что делать при двойном submit формы матча?”
Нужно смотреть:

- `VALIDATION_RULES.md` — anti-duplicate правила
- `TASKS.md` — acceptance criteria этапа
- `AGENTS.md` — дисциплина реализации без лишнего расширения scope

### Пример 4: “Можно ли заменить PostgreSQL на другую БД?”
Нужно смотреть:

- `TECH_STACK.md` — зафиксированный стек
- `AGENTS.md` — запрет на самовольную замену технических решений

## 6. Правило при конфликте документов

Если документы конфликтуют между собой, использовать следующий приоритет:

1. `PRD.md`
2. `DOMAIN_MODEL.md`
3. `RBAC.md`
4. `TASKS.md`
5. `TECH_STACK.md`
6. `VALIDATION_RULES.md`
7. `AGENTS.md`

### Как это интерпретировать

- `PRD.md` имеет приоритет над всеми остальными документами в вопросах продукта
- `DOMAIN_MODEL.md` имеет приоритет над нижестоящими документами в вопросах сущностей и структуры данных
- `RBAC.md` имеет приоритет над нижестоящими документами в вопросах ролей и доступа
- `TASKS.md` имеет приоритет над нижестоящими документами в вопросах порядка реализации
- `TECH_STACK.md` имеет приоритет над `VALIDATION_RULES.md` и `AGENTS.md` в вопросах стека и инфраструктуры
- `VALIDATION_RULES.md` имеет приоритет над `AGENTS.md` в вопросах валидаций и негативных сценариев
- `AGENTS.md` не переопределяет продукт, модель, роли, этапы, стек или валидации

---

## 7. Как пользоваться документацией перед любой задачей

Перед началом любой новой задачи нужно:

1. открыть `DOC_INDEX.md`
2. определить, к какому слою относится задача
3. открыть документы в обязательном порядке
4. отдельно зафиксировать:
   - что входит в задачу
   - что не входит в задачу
   - какие файлы можно менять
   - какие правила являются критичными
5. только после этого начинать реализацию

## 9. Практическое правило для Codex

Перед любой задачей:

1. сначала открыть `DOC_INDEX.md`
2. затем открыть остальные документы в порядке, указанном выше
3. определить, какие из них действительно нужны для текущей задачи
4. не додумывать правила, если они уже описаны в документации
5. не искать ответ в неправильном документе

## 10. Краткий итог

`DOC_INDEX.md` — это **навигационная карта проекта**.

Главный смысл документа:

- сначала открыть этот файл
- затем читать документы в правильном порядке
- искать ответ в документе, который отвечает именно за этот тип вопроса
- не смешивать продукт, модель данных, роли, этапы, стек, валидации и правила работы Codex в один слой
