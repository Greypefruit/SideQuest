# DOMAIN_MODEL.md — SideQuest Release 1

## 1. Назначение документа

Этот документ описывает **доменную модель Release 1** проекта **SideQuest**:

- какие сущности существуют в системе
- какие связи между сущностями допустимы
- какие поля являются обязательными
- какие ограничения и инварианты должны соблюдаться
- какие сущности не нужно добавлять в Release 1

Этот документ является главным источником истины по структуре данных приложения.

Если реализация противоречит `PRD.md`, приоритет имеет `PRD.md`.  
Если в реализации появляется неоднозначность по таблицам, связям или полям, приоритет имеет этот документ.

---

## 2. Границы доменной модели Release 1

### 2.1 Что моделируется в Release 1

Release 1 моделирует:

- пользователей системы
- роли пользователей
- типы активностей
- участие пользователя в активности
- рейтинг участника по активности
- обычные матчи
- турниры
- участников турниров
- матчи внутри турнирной сетки

### 2.2 Что не моделируется отдельными сущностями в Release 1

В Release 1 не нужно создавать отдельные сущности или таблицы для:

- истории как отдельного объекта
- уведомлений
- комментариев
- чата
- достижений
- аватаров
- аудита
- логов изменений
- споров по матчам
- подтверждения результата вторым участником
- турнирных сезонов
- альтернативных рейтингов
- self-registration в турнир
- переключения ролей
- нескольких ролей у одного пользователя
- отдельных сущностей для каждой активности
- отдельной сущности `users`, если уже есть `profiles`
- отдельной сущности `match_results`, если результат хранится внутри матчей
- отдельной сущности `competition_match_results`, если результат хранится внутри турнирных матчей
- отдельной сущности `history`
- отдельной сущности `sessions`
- отдельной сущности `events`

### 2.3 Важное ограничение Release 1

UI Release 1 работает только для **настольного тенниса**.  
При этом доменная модель должна быть **универсальной** и пригодной для расширения на другие активности в будущем.

## 3. Общие принципы моделирования

### 3.1 Общий принцип

Система строится вокруг универсальных сущностей платформы, а не вокруг одной конкретной игры.

### 3.2 Что это означает practically

- пользователь системы — это `profile`
- участие пользователя в конкретной активности — это `participant`
- рейтинг существует не “вообще у пользователя”, а у участника в рамках активности
- обычные матчи привязаны к участникам активности
- турниры привязаны к активности
- участники турнира — это участники активности
- турнирные матчи не смешиваются с обычными матчами

### 3.3 Общие технические соглашения

Для всех доменных таблиц Release 1:

- использовать **UUID** как первичный ключ
- использовать явные timestamps:
  - `created_at`
  - `updated_at`, если запись предполагает изменение
- использовать английские названия таблиц и полей
- использовать русские названия только в пользовательском интерфейсе

### 3.4 Общая политика удаления

В Release 1 исторические данные не должны свободно удаляться через UI.

Общие правила:

- завершенные обычные матчи не удаляются через UI
- завершенные турнирные матчи не удаляются через UI
- завершенные турниры не удаляются через UI
- пользователь деактивируется, а не удаляется физически через UI

## 5. Сущность `profiles`

### 5.1 Назначение

`profiles` — это пользователи приложения.

Профиль представляет:

- личность пользователя в системе
- email пользователя
- системную роль пользователя
- активность пользователя в системе

`profiles` — единственная доменная сущность пользователя в Release 1.  
Не нужно создавать дополнительную таблицу `users` для доменной модели приложения.

### 5.2 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `auth_user_id` — string/UUID, unique, not null
- `email` — string, unique, not null
- `display_name` — string, not null
- `role` — enum/string, not null
- `is_active` — boolean, not null, default `true`
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 5.3 Значение полей

#### `auth_user_id`
Связь с системой аутентификации.

Важно:
- доменная модель не обязана хранить OTP-коды
- в доменной модели достаточно хранить идентификатор пользователя из auth subsystem

#### `email`
Основной идентификатор пользователя в бизнес-логике Release 1.

Ограничения:
- уникален
не nullable

#### Правило email и входа в Release 1

В Release 1 поле `email` используется как email для входа пользователя в систему.

Обязательные правила:

- email может принадлежать любому домену
- email не ограничивается корпоративными доменами
- вход выполняется по `email + одноразовый код подтверждения`
- профиль создается при первом успешном входе пользователя
- доменная модель не требует отдельной сущности whitelist или invite-list для Release 1

Следствие Release 1:

- наличие корректного email само по себе не означает существование профиля
- профиль появляется только после первого успешного подтвержденного входа
- один профиль использует один email для входа

#### `display_name`
Отображаемое имя пользователя в UI.

Правило Release 1:
- если отдельное имя еще не получено, значение может быть автоматически сгенерировано на основе email при первом входе


#### `role`
Системная роль пользователя.

Допустимые значения Release 1:

- `player`
- `organizer`
- `admin`

#### `is_active`
Флаг активности пользователя.

Назначение:
- активный пользователь может работать в системе
- деактивированный пользователь остается в данных, но считается неактивным

### 5.4 Инварианты

- у каждого профиля ровно одна роль
- `auth_user_id` уникален
- `email` уникален
- `role` принимает только допустимые значения
- `is_active` обязателен
- новый пользователь автоматически получает `player`
- email может принадлежать любому домену
- профиль создается при первом успешном входе через `email + OTP`

### 5.5 Связи

`profiles` связана со следующими сущностями:

- один `profile` → много `participants`
- один `profile` → много созданных `matches` через `created_by_profile_id`
- один `profile` → много созданных `competitions` через `created_by_profile_id`
- один `profile` → много `competition_participants` через `added_by_profile_id`
один `profile` → много `competition_matches` через `reported_by_profile_id`

### 5.6 Деактивация пользователя

В Release 1 деактивированный пользователь остается в доменных данных, но считается недоступным для использования системы.

Это означает:

- профиль не удаляется физически
- связанные `participants`, матчи, турниры и исторические записи сохраняются
- деактивированный пользователь не должен получать доступ к приложению после успешной проверки аутентификации
- деактивированный пользователь не должен выполнять новые действия в системе

## 6. Сущность `activity_types`

### 6.1 Назначение

`activity_types` — справочник типов активностей, который нужен для универсальной модели платформы.

Примеры возможных активностей:

- table tennis
- billiards
- darts
- foosball
- online games

В Release 1 UI работает только для настольного тенниса, но сущность должна быть универсальной.

### 6.2 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `code` — string, unique, not null
- `name_ru` — string, not null
- `is_active` — boolean, not null, default `true`
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 6.3 Значение полей

#### `code`
Стабильный технический код активности.

Пример:
- `table_tennis`

#### `name_ru`
Русское пользовательское название активности.

Пример:
- `Настольный теннис`

#### `is_active`
Флаг активности типа активности.

UI-управление этим справочником в Release 1 не требуется.

### 6.4 Seed data Release 1

В Release 1 в таблице `activity_types` должна существовать как минимум одна запись:

- `code = table_tennis`
- `name_ru = Настольный теннис`

### 6.5 Инварианты

- `code` уникален
- `name_ru` не должен быть пустым
- хотя бы один activity type должен существовать в seed data Release 1

### 6.6 Связи

- один `activity_type` → много `participants`
- один `activity_type` → много `matches`
один `activity_type` → много `competitions`

## Контекст текущей активности Release 1

Несмотря на то, что доменная модель Release 1 проектируется как расширяемая для нескольких активностей, пользовательские экраны Release 1 работают только в контексте одной активности:

- `table_tennis`
- **Настольный теннис**

Это правило должно учитываться не только на уровне UI, но и на уровне доменной модели и серверных запросов.

### Доменные следствия Release 1

Для Release 1 необходимо считать, что:

- `rankings` всегда интерпретируются в рамках конкретной активности
- `matches` всегда принадлежат конкретной активности
- `competitions` всегда принадлежат конкретной активности
- `participants` существуют только в связке `profile + activity_type`
- пользовательские списки и агрегаты не должны смешивать данные разных активностей

### Источник activity scope

Источником activity scope в Release 1 являются:

- `participants.activity_type_id`
- `matches.activity_type_id`
- `competitions.activity_type_id`

Все пользовательские выборки для основных экранов должны опираться на это ограничение.

### Обязательное правило выборки данных

Даже если в Release 1 интерфейс не показывает отдельный фильтр активности, доменная логика должна считать, что:

- рейтинг выбирается по текущей активности
- последние матчи выбираются по текущей активности
- турниры выбираются по текущей активности
- участие пользователя и его статистика интерпретируются по текущей активности

Недопустимо для Release 1:

- строить глобальный рейтинг сразу по всем активностям
- смешивать матчи разных активностей в одной пользовательской выборке
- смешивать турниры разных активностей в одной пользовательской выборке
- трактовать `participants`, `rankings`, `matches` и `competitions` как объекты без activity context

### Ограничение Release 1

В Release 1 UI может не иметь отдельного переключателя активности, поскольку активность фиксирована и равна `table_tennis`.

Однако это не отменяет обязательного доменного правила:
все пользовательские выборки должны быть activity-scoped уже в Release 1.

### Требование к будущему расширению

При добавлении новых активностей после Release 1 расширение должно происходить через смену текущего activity context, а не через переписывание доменной модели или глобальных пользовательских экранов.

Поэтому уже в Release 1 доменная модель должна рассматривать:

- рейтинг как рейтинг в рамках активности
- матч как матч в рамках активности
- турнир как турнир в рамках активности
- участие пользователя как участие в конкретной активности

## 7. Сущность `participants`

### 7.1 Назначение

`participants` — это участие профиля в конкретной активности.

Это ключевая сущность универсальной модели.

Важно:
- `profile` — это человек в системе
- `participant` — это этот же человек внутри конкретной активности

### 7.2 Зачем нужна отдельная сущность `participants`

Она нужна, чтобы:

- один пользователь мог участвовать в нескольких активностях в будущем
- рейтинг был привязан к активности
- матчи были привязаны к участнику активности, а не к абстрактному пользователю

### 7.3 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `profile_id` — UUID, FK → `profiles.id`, not null
- `activity_type_id` — UUID, FK → `activity_types.id`, not null
- `is_active` — boolean, not null, default `true`
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 7.4 Уникальность

Должно существовать ограничение:

- `UNIQUE(profile_id, activity_type_id)`

Это значит:
- у одного пользователя не может быть двух участников для одной и той же активности

### 7.5 Поведение Release 1

При первом успешном входе нового пользователя система должна автоматически:

1. создать `profile`
2. создать `participant` для default activity `table_tennis`
3. создать запись `ranking` для этого participant

Это правило применяется для любого корректного email, с которым пользователь впервые успешно входит в систему через `email + OTP`.

### 7.6 Инварианты

- один `profile` может иметь много `participants`
- в рамках одной активности у профиля только один `participant`
- `participant` нельзя создать без существующего `profile`
- `participant` нельзя создать без существующего `activity_type`

### 7.7 Связи

- много `participants` → один `profile`
- много `participants` → один `activity_type`
- один `participant` → одна запись в `rankings`
- один `participant` → много `matches`
- один `participant` → много `competition_participants`
- один `participant` → много `competition_matches`

## 8. Сущность `rankings`

### 8.1 Назначение

`rankings` хранит **текущий снимок рейтинга** участника в рамках одной активности.

Это не история всех пересчетов.  
Это актуальное состояние рейтинга, которое используется для страницы рейтинга и профиля.

### 8.2 Важное правило

В Release 1 не нужно создавать отдельную таблицу истории рейтинга, если этого нет в scope текущей задачи.

Источником рейтинговых изменений являются:

- `matches`
- `competition_matches`

`rankings` — это текущий snapshot.

### 8.3 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `participant_id` — UUID, FK → `participants.id`, unique, not null
- `rating` — integer, not null
- `matches_played` — integer, not null, default `0`
- `wins` — integer, not null, default `0`
- `losses` — integer, not null, default `0`
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 8.4 Правила Release 1

- у каждого `participant` ровно одна запись `ranking`
- стартовый рейтинг нового участника: `1000`
- Elo — единственный алгоритм рейтинга Release 1
- `K-factor = 32`
- рейтинг обновляется только после завершенного матча
- обычные и турнирные матчи влияют на один и тот же рейтинг
- `bye` не влияет на рейтинг
- ничья не поддерживается

### 8.5 Инварианты

- `participant_id` уникален
- рейтинг не существует без participant
- `matches_played`, `wins`, `losses` не могут быть отрицательными

### 8.6 Связи

- один `ranking` → один `participant`
- один `participant` → одна запись `ranking`

## 9. Сущность `matches`

### 9.1 Назначение

`matches` — это обычные матчи вне турниров.

Эта таблица предназначена только для матчей, которые:

- не являются частью турнирной сетки
- создаются сразу с финальным результатом

### 9.2 Важное правило Release 1

В Release 1 каждая запись в `matches` — это уже завершенный обычный матч.

То есть:

- отдельная стадия планирования не нужна
- статусы `planned` / `in_progress` не нужны
- запись без результата не нужна

### 9.3 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `activity_type_id` — UUID, FK → `activity_types.id`, not null
- `participant1_id` — UUID, FK → `participants.id`, not null
- `participant2_id` — UUID, FK → `participants.id`, not null
- `match_format` — enum/string, not null
- `participant1_score` — integer, not null
- `participant2_score` — integer, not null
- `winner_participant_id` — UUID, FK → `participants.id`, not null
- `created_by_profile_id` — UUID, FK → `profiles.id`, not null
- `played_at` — timestamp, not null
- `created_at` — timestamp, not null

### 9.4 Значение полей

#### `activity_type_id`
Указывает, к какой активности относится матч.

#### `participant1_id` и `participant2_id`
Участники матча.

#### `match_format`
Формат матча.

Допустимые значения Release 1:

- `BO1`
- `BO3`
- `BO5`

#### `participant1_score` и `participant2_score`
Итоговый счет матча по партиям.

Важно:
- это итоговый счет, а не поочковый протокол партий
- в Release 1 хранится только финальный результат, достаточный для UI и Elo

#### `winner_participant_id`
Победитель матча.

В Release 1:
- ничья не поддерживается
- победитель обязателен

#### `created_by_profile_id`
Кто создал запись о матче.

Это поле ссылается на `profiles`, а не на `participants`, потому что действие совершает пользователь системы.

#### `played_at`
Когда матч был сыгран или зафиксирован как сыгранный.

### 9.5 Инварианты

- матч всегда `1 на 1`
- `participant1_id != participant2_id`
- формат матча обязателен
- `match_format` принимает только `BO1`, `BO3`, `BO5`
- оба участника должны принадлежать одной и той же активности
- `winner_participant_id` должен быть либо `participant1_id`, либо `participant2_id`
- `participant1_score` и `participant2_score` должны быть целыми неотрицательными числами
- ничья в Release 1 не поддерживается
- матч не может существовать без финального результата
- матч не может существовать без победителя

### 9.6 Допустимые результаты по формату

Если `match_format = BO1`, допустимы только:

- `1:0`
- `0:1`

Если `match_format = BO3`, допустимы только:

- `2:0`
- `2:1`
- `0:2`
- `1:2`

Если `match_format = BO5`, допустимы только:

- `3:0`
- `3:1`
- `3:2`
- `0:3`
- `1:3`
- `2:3`

Любой другой итоговый счет должен считаться невалидным для Release 1.

### 9.7 Чего не нужно делать в `matches`

В Release 1 не нужно:

- поле статуса `planned` / `pending` / `in_progress`
- отдельную таблицу `match_results`
- поле подтверждения результата вторым игроком
- поле спорного результата
- подробный протокол по очкам каждой партии

### 9.8 Связи

- много `matches` → один `activity_type`
- много `matches` → два `participants`
- много `matches` → один `profile` через `created_by_profile_id`

## 10. Сущность `competitions`

### 10.1 Назначение

`competitions` — это турниры/соревнования.

В Release 1 поддерживается только один формат турнира:

- `single_elimination`

### 10.2 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `activity_type_id` — UUID, FK → `activity_types.id`, not null
- `title` — string, not null
- `format` — enum/string, not null
- `match_format` — enum/string, not null
`status` — enum/string, not null
`scheduled_at` — timestamp, nullable
`location` — string, nullable
- `created_by_profile_id` — UUID, FK → `profiles.id`, not null
- `started_at` — timestamp, nullable
- `completed_at` — timestamp, nullable
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 10.3 Допустимые значения

#### `format`
В Release 1 допустимо только:

- `single_elimination`

#### `match_format`
В Release 1 допустимы только:

- `BO1`
- `BO3`
- `BO5`

#### `status`
В Release 1 допустимы только:

- `draft`
- `in_progress`
- `completed`
- `cancelled`

### 10.4 Значение статусов

#### `draft`
Турнир создан, состав участников можно редактировать, сетка еще не запущена.

#### `in_progress`
Сетка сгенерирована, турнир идет.

#### `completed`
Турнир завершен, финальный победитель определен.

#### `cancelled`
Турнир отменен.

### 10.4.1 Дополнительные поля планирования

#### `scheduled_at`
Опциональная дата и время проведения турнира.

Правила Release 1:

- поле не является обязательным для создания турнира
- турнир может быть создан как `draft` без заполненного `scheduled_at`
- значение может быть добавлено или изменено до старта турнира

#### `location`
Опциональная локация проведения турнира.

Правила Release 1:

- поле не является обязательным для создания турнира
- турнир может быть создан как `draft` без заполненной локации
значение может быть добавлено или изменено до старта турнира


### 10.5 Инварианты

- турнир относится ровно к одной активности
- турнир имеет один формат
- в Release 1 `format` всегда `single_elimination`
- `match_format` обязателен
- `match_format` принимает только `BO1`, `BO3`, `BO5`
- турнир может быть создан в статусе `draft` без участников
- участники добавляются после создания турнира на этапе `draft`
- `scheduled_at` и `location` являются опциональными
- после перехода в `in_progress` состав участников нельзя менять
- после перехода в `in_progress` `match_format` менять нельзя
- после перехода в `in_progress` поля планирования не должны менять фактическую историю турнира как активного объекта без отдельного решения
- завершенный турнир не редактируется как активный

### 10.6 Владение турниром

Поле `created_by_profile_id` является источником истины для владельца турнира в Release 1.

Правило владения:

- владелец турнира определяется через `competitions.created_by_profile_id`
- пользователь с ролью `organizer` может управлять только турнирами, где `created_by_profile_id` совпадает с его `profile.id`
пользователь с ролью `admin` может управлять любым турниром

### 10.6.1 Редактирование турнира в статусе `draft`

Пока турнир находится в статусе `draft`, владелец турнира или `admin` могут изменять:

- `title`
- `match_format`
- `scheduled_at`
- `location`

В Release 1 не предполагается изменение:

- `activity_type_id`
- `format` (`single_elimination`)

После перехода турнира в `in_progress` изменение этих полей должно ограничиваться правилами Release 1 и не должно нарушать уже созданную турнирную структуру.

### 10.7 Связи

- много `competitions` → один `activity_type`
- много `competitions` → один `profile` через `created_by_profile_id`
- один `competition` → много `competition_participants`
- один `competition` → много `competition_matches`

## 11. Сущность `competition_participants`

### 11.1 Назначение

`competition_participants` связывает турнир и участников активности.

Эта сущность отвечает на вопрос:

**“Кто участвует в данном турнире?”**

### 11.2 Почему нужна отдельная таблица

Она нужна, потому что:

- не каждый participant обязательно участвует в каждом tournament
- один participant может участвовать в нескольких турнирах
- один tournament имеет много участников

### 11.3 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `competition_id` — UUID, FK → `competitions.id`, not null
- `participant_id` — UUID, FK → `participants.id`, not null
- `added_by_profile_id` — UUID, FK → `profiles.id`, not null
- `created_at` — timestamp, not null

### 11.4 Уникальность

Должно существовать ограничение:

- `UNIQUE(competition_id, participant_id)`

Это значит:
- один и тот же участник не может быть добавлен в один и тот же турнир дважды

### 11.5 Инварианты

- `participant` должен принадлежать той же активности, что и `competition`
- участник турнира не может существовать без существующего турнира
- участники турнира добавляются только после создания турнира на этапе `draft`
- self-registration в Release 1 не поддерживается

### 11.6 Связи

- много `competition_participants` → один `competition`
- много `competition_participants` → один `participant`
- много `competition_participants` → один `profile` через `added_by_profile_id`

## 12. Сущность `competition_matches`

### 12.1 Назначение

`competition_matches` хранит матчи внутри турнирной сетки.

Эта сущность не должна смешиваться с `matches`.

Разделение обязательное:

- `matches` — обычные матчи вне турниров
- `competition_matches` — матчи внутри турнирной сетки

### 12.2 Основные требования Release 1

Турнирный матч должен позволять:

- хранить двух участников
- хранить итоговый результат
- хранить победителя
- знать позицию в сетке
- продвигать победителя дальше по сетке

### 12.3 Поля

Рекомендуемый состав полей:

- `id` — UUID, primary key
- `competition_id` — UUID, FK → `competitions.id`, not null
- `round_number` — integer, not null
- `match_number` — integer, not null
- `slot1_participant_id` — UUID, FK → `participants.id`, nullable
- `slot2_participant_id` — UUID, FK → `participants.id`, nullable
- `slot1_score` — integer, nullable
- `slot2_score` — integer, nullable
- `winner_participant_id` — UUID, FK → `participants.id`, nullable
- `status` — enum/string, not null
- `resolution_type` — enum/string, nullable
- `next_match_id` — UUID, FK → `competition_matches.id`, nullable
- `next_match_slot` — integer, nullable
- `reported_by_profile_id` — UUID, FK → `profiles.id`, nullable
- `completed_at` — timestamp, nullable
- `created_at` — timestamp, not null
- `updated_at` — timestamp, not null

### 12.4 Значение полей

#### `round_number`
Номер раунда в сетке.

#### `match_number`
Порядковый номер матча внутри раунда.

#### `slot1_participant_id` и `slot2_participant_id`
Участники конкретного турнирного матча.

Для будущих раундов один или оба слота могут быть пустыми до определения победителей предыдущих матчей.

#### `status`
Рекомендуемые значения Release 1:

- `pending`
- `completed`

#### `resolution_type`
Рекомендуемые значения Release 1:

- `played`
- `bye`

Назначение:
- `played` — обычный сыгранный турнирный матч
- `bye` — автоматический проход одного участника

#### `next_match_id`
Матч следующего раунда, куда должен быть передан победитель.

#### `next_match_slot`
В какой слот следующего матча должен быть помещен победитель.

Допустимые значения:

- `1`
- `2`

#### `reported_by_profile_id`
Кто внес результат турнирного матча.

### 12.5 Инварианты

#### Для всех турнирных матчей

- матч принадлежит ровно одному турниру
- участники матча должны принадлежать той же активности, что и турнир
- после завершения матча победитель должен быть определен однозначно
- ничья не поддерживается

#### Для `resolution_type = played`

- оба участника должны быть заполнены
- победитель должен быть одним из двух участников
- счет должен быть указан
- итоговый счет должен соответствовать `competition.match_format`
- `slot1_score` и `slot2_score` должны быть целыми неотрицательными числами

#### Для `resolution_type = bye`

- должен быть только один реальный участник, который проходит дальше
- победитель должен совпадать с этим участником
- Elo для `bye` не пересчитывается

### 12.6 Допустимые результаты по формату турнира

Для `competition.match_format` применяются те же ограничения по счету, что и для обычных матчей:

- `BO1` → только `1:0` или `0:1`
- `BO3` → только `2:0`, `2:1`, `0:2`, `1:2`
- `BO5` → только `3:0`, `3:1`, `3:2`, `0:3`, `1:3`, `2:3`

Турнирный матч нельзя завершить результатом, который не соответствует формату турнира.

### 12.7 Важное правило по bracket logic

Для продвижения победителя в следующий раунд Release 1 должно быть достаточно полей:

- `next_match_id`
- `next_match_slot`

Не нужно строить более сложную универсальную модель bracket engine для Release 1.

### 12.8 Чего не нужно делать в `competition_matches`

В Release 1 не нужно:

- отдельную таблицу `competition_match_results`
- подробный покадровый или поочковый протокол
- статусы спорного матча
- подтверждение результата участниками
- ручную сложную перестройку сетки после старта

### 12.9 Связи

- много `competition_matches` → один `competition`
- каждая запись содержит до двух `participants` и опционального победителя
- один `competition_match` → опционально один следующий `competition_match`
- много `competition_matches` → один `profile` через `reported_by_profile_id`

### `profiles`
- 1 → N `participants`

- 1 → N `competitions` (creator)
- 1 → N `competition_participants` (added_by)

### `activity_types`
- 1 → N `participants`
- 1 → N `matches`
- 1 → N `competitions`

- N → 1 `profiles`
- N → 1 `activity_types`
- 1 → 1 `rankings`
- 1 → N `matches`
- 1 → N `competition_participants`
- 1 → N `competition_matches`

### `rankings`
- 1 → 1 `participants`

### `matches`
- N → 1 `activity_types`
- N → 1 `profiles` (creator)
- каждая запись содержит 2 participant references и 1 winner reference

### `competitions`
- N → 1 `activity_types`
- N → 1 `profiles` (creator)

- N → 1 `competitions`

- N → 1 `profiles` (added_by)

### `competition_matches`
- N → 1 `competitions`
- каждая запись содержит до 2 participant references и optional winner reference
- optional self-reference на следующий матч

## 14. Создание доменных записей в основных сценариях

### 14.1 Первый вход нового пользователя

При первом успешном входе нового пользователя должны быть созданы:

1. `profile`
2. `participant` для default activity `table_tennis`
3. `ranking` со стартовым Elo `1000`

### 14.1.1 Повторный вход существующего пользователя

При повторном успешном входе пользователя система не должна создавать дублирующие доменные записи.

Это означает:

- существующий `profile` переиспользуется
- существующий `participant` для `table_tennis` переиспользуется
- существующий `ranking` переиспользуется
- повторный вход не создает второй профиль для того же email

### 14.2 Создание обычного матча

При создании обычного матча должны быть:

1. выбрана активность
2. выбраны два участника одной активности
3. выбран `match_format`
4. указан итоговый счет
5. указан победитель
6. создана запись в `matches`
7. обновлены `rankings` обоих участников

### 14.3 Создание турнира

При создании турнира должны быть:

1. создан `competition`
2. выбран `match_format`
3. при необходимости сохранены `scheduled_at` и `location`
4. турнир сохранен в статусе `draft`

Добавление участников выполняется после создания турнира на странице турнира в статусе `draft`.

После добавления участников и запуска генерации сетки должны быть:

1. созданы `competition_participants`
2. созданы `competition_matches`
3. турнир переведен в `in_progress`

### 14.4 Внесение результата турнирного матча

При завершении турнирного матча должны быть:

1. обновлена запись `competition_match`
2. определен победитель
3. победитель помещен в `next_match`
4. если это не `bye`, обновлен Elo в `rankings`
