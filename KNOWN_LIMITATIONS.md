# KNOWN_LIMITATIONS.md — SideQuest Release 1

Список реальных ограничений Release 1, выявленных при аудите кодовой базы. Это осознанные границы MVP, а не открытые баги. Critical / High проблем не обнаружено.

---

## Матчи и идемпотентность

- **Нет server-side идемпотентности при создании обычного матча.** Защита от повторной отправки реализована только на UI (блокировка кнопки на время запроса) + транзакция. При повторной доставке запроса в обход UI (сетевой ретрай, прямой повторный вызов action) теоретически возможны дубль матча и двойной Elo-update. Соответствует допущению `VALIDATION_RULES §7.9`, где UI-блокировка указана как допустимый способ.

## Турниры

- **Нет rollback Elo после отмены турнира.** Поэтому отмена турнира разрешена только пока в нём нет сыгранных (`played`, завершённых) матчей, уже повлиявших на рейтинг. После первого сыгранного матча отмена блокируется. Это сознательная защита от рассогласования рейтинга (`VALIDATION_RULES §8.12`).
- **Турнир переходит в статус «идёт» только по наступлении `scheduled_at`.** При создании турнира дата и время обязательны и должны быть в будущем. Сетку можно сгенерировать заранее (статус «готов»), но ввод результатов становится доступен только когда наступило запланированное время. Режима «начать прямо сейчас» нет.
- **Поддерживается только формат `single_elimination`.** Round-robin, групповые этапы, double-elimination и т.п. в Release 1 отсутствуют.

## Рейтинг

- **В таблицу рейтинга попадают все провизиненные участники**, включая тех, кто ещё не сыграл ни одного матча (рейтинг 1000, 0 матчей). Отдельной фильтрации «несыгравших» нет.
- **Рейтинг нельзя редактировать вручную через обычный UI** — он меняется только как следствие матчей. Это ограничение по дизайну (нет административного ручного редактирования Elo).

## Авторизация и доступ

- **OTP-код состоит из 4 цифр** (10000 комбинаций), до 5 попыток на код. Достаточно для внутреннего MVP, но энтропия ниже, чем у 6-значного кода. Повторный запрос кода инвалидирует предыдущий и ограничен cooldown'ом 30 секунд.
- **В dev-режиме без настроенного SMTP код OTP выводится в консоль сервера** (fallback для разработки). В production при неполной SMTP-конфигурации отправка письма завершается ошибкой — SMTP обязателен для эксплуатации.
- **Единственный способ входа — email + OTP.** Пароли, OAuth, SSO и т.п. не поддерживаются. Деактивированный пользователь не может войти и не выполняет новые действия.

## Активности

- **Release 1 работает только с одной активностью — настольный теннис (`table_tennis`).** Доменная модель расширяема под несколько активностей, но UI и пользовательские выборки зафиксированы на текущей активности. Переключателя активностей в интерфейсе нет.

## Данные и инфраструктура

- **Seed-данные применяются вручную.** `drizzle-kit migrate` не выполняет файлы из `drizzle-mysql/seeds/`. Запись активности `table_tennis` обязательна для работы провижининга и должна быть загружена отдельно (см. `RELEASE.md`).
- **Локальная БД рассчитана на запуск через Docker Compose** с дефолтными учётными данными из `compose.yaml`. Для production учётные данные и параметры подключения должны быть заменены через окружение.

## Тестирование

- **Автоматических тестов в репозитории нет** (в `package.json` нет скрипта `test`). Контроль качества Release 1 опирается на `npm run lint`, `npx tsc --noEmit`, `npm run build` и ручную проверку сценариев.
