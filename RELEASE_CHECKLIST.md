# RELEASE_CHECKLIST.md — SideQuest Release 1

Чеклист выпуска и проверки эксплуатационной готовности. Отмечать пункты последовательно.

---

## Перед запуском

- [ ] Ветка релиза синхронизирована с `origin`, рабочее дерево чистое (`git status`).
- [ ] `npm run lint` — зелёный.
- [ ] `npx tsc --noEmit` — зелёный.
- [ ] `npm run build` — зелёный.
- [ ] `web/.env.local` (или production-окружение) заполнен: `DATABASE_URL`, `AUTH_SESSION_SECRET`.
- [ ] `AUTH_SESSION_SECRET` — длинная случайная строка (не dev-значение).
- [ ] SMTP-переменные заданы полностью (`SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASSWORD`, `OTP_EMAIL_FROM`).
- [ ] Секреты не закоммичены в репозиторий.
- [ ] Сделан backup текущей БД (`mysqldump`), файл сохранён вне репозитория.
- [ ] Миграции применены (`npx drizzle-kit migrate`).
- [ ] Seed-активность `table_tennis` загружена.

## После деплоя

- [ ] Приложение поднимается без ошибок (`npm run start` / процесс в проде живой).
- [ ] Главная страница и страница входа открываются.
- [ ] Все внутренние маршруты требуют авторизации (неавторизованный редиректится на вход).
- [ ] В логах нет ошибок подключения к БД (`DATABASE_URL is not set` / connection errors).
- [ ] В логах нет ошибок SMTP при первом запросе OTP.

## Проверка OTP

- [ ] Запрос кода по корректному email проходит, письмо приходит.
- [ ] Неверный код отклоняется с сообщением «Неверный код».
- [ ] Повторный запрос кода чаще 30 секунд возвращает ограничение.
- [ ] Просроченный код (>10 минут) не пускает в систему.
- [ ] После 5 неверных попыток код становится недействительным.
- [ ] Первый успешный вход создаёт профиль с ролью `player`.
- [ ] Повторный вход существующего пользователя работает.
- [ ] Деактивированный пользователь не может войти («Ваш доступ отключен…»).

## Проверка БД

- [ ] Подключение к MySQL работает (контейнер/инстанс доступен).
- [ ] Таблицы созданы согласно миграции `0000_pale_inertia`.
- [ ] Запись `activity_types` с кодом `table_tennis` присутствует.
- [ ] Создание матча/турнира действительно пишет данные в БД.
- [ ] Backup и restore проверены на тестовом дампе (восстановление поднимается).

## Проверка турниров

- [ ] Организатор/админ может создать турнир (player — не может).
- [ ] Добавление и удаление участников до старта работает; дубль участника отклоняется.
- [ ] Лимит участников соблюдается.
- [ ] Генерация сетки возможна от 2 участников; повторная генерация отклоняется.
- [ ] Bye-матчи проставляются корректно, не требуют ручного ввода счёта.
- [ ] Ввод результата доступен только после наступления `scheduled_at` (статус «идёт»).
- [ ] Победитель продвигается в следующий матч; финал завершает турнир.
- [ ] Отмена турнира блокируется при наличии сыгранных матчей.
- [ ] Организатор не может управлять чужим турниром; админ — может.

## Проверка рейтинга

- [ ] После обычного матча рейтинг обоих участников меняется (сумма дельт = 0).
- [ ] После турнирного матча рейтинг участников обновляется.
- [ ] Таблица рейтинга отсортирована (рейтинг ↓, далее матчи/победы/поражения, имя, email).
- [ ] Позиция и Elo текущего пользователя отображаются корректно.
- [ ] Один завершённый матч влияет на Elo только один раз (нет двойного пересчёта).

## Проверка администратора

- [ ] Раздел пользователей доступен только роли `admin`.
- [ ] Смена роли работает (allowlist: player/organizer/admin).
- [ ] Активация/деактивация пользователя работает.
- [ ] Последнего активного администратора нельзя понизить или деактивировать.
- [ ] Прямой вызов admin-действия не-админом отклоняется на сервере.

## План отката

- [ ] Зафиксирован предыдущий рабочий коммит/тег для быстрого возврата кода.
- [ ] Есть свежий backup БД, сделанный до деплоя (раздел «Перед запуском»).
- [ ] Процедура отката кода: вернуть предыдущий релиз (redeploy предыдущего коммита/тега).
- [ ] Процедура отката БД: восстановить дамп из backup (см. «Восстановление» в `RELEASE.md`).
- [ ] Если миграция оказалась проблемной — восстановить БД из backup, сделанного до миграции (rollback миграций вперёд не предусмотрен).
- [ ] После отката повторить базовые проверки (вход по OTP, открытие главной, доступность БД).
- [ ] Зафиксированы контакты ответственных за инфраструктуру/почтовый сервис на время релиза.
