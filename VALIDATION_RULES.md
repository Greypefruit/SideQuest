# VALIDATION_RULES.md — SideQuest Release 1

## 1. Назначение документа

Этот документ описывает **правила валидации, негативные сценарии и защиту от невалидных состояний** для проекта **SideQuest Release 1**.

Документ определяет:

- что именно должно валидироваться
- на каком уровне должна происходить валидация
- какие пользовательские действия должны блокироваться
- какие невалидные состояния данных не должны попадать в систему
- какие anti-duplicate и anti-race правила обязательны

Этот документ не заменяет:

- `PRD.md` — продуктовые требования
- `DOMAIN_MODEL.md` — доменная модель и инварианты данных
- `RBAC.md` — роли и права доступа
- `TASKS.md` — порядок реализации
- `TECH_STACK.md` — стек и инфраструктуру
- `AGENTS.md` — правила работы Codex

Если возникает конфликт:
- `PRD.md` имеет приоритет над этим документом
- `DOMAIN_MODEL.md` имеет приоритет над структурой данных
- `RBAC.md` имеет приоритет над доступами

## 2. Общие принципы валидации

### 2.1 Валидация должна происходить на трех уровнях

Там, где это применимо, валидация должна существовать на трех уровнях:

1. **UI level** — чтобы пользователь не вводил очевидно неверные данные
2. **Server level** — как основной обязательный уровень истины
3. **DB level** — для критичных инвариантов данных, если их можно выразить ограничениями

### 2.2 Источник истины

UI помогает пользователю, но не является источником истины.

Обязательное правило:

- даже если UI уже ограничивает действие, сервер обязан повторно проверить все критичные условия

### 2.3 Нормализация перед валидацией

Перед проверкой входные данные должны быть приведены к нормальной форме там, где это уместно.

Примеры:

- email → `trim` + `lowercase`
- текстовые поля → `trim`
- числовые поля → parse + strict integer validation

### 2.4 Нельзя доверять только типу поля в браузере

Нельзя считать, что `input type="number"` гарантирует корректность данных.

Сервер должен отвергать:

- строки вместо чисел
- дробные значения
- `NaN`
- `Infinity`
- пустые значения там, где поле обязательно
- значения вне допустимого диапазона

### 2.5 Правило идемпотентности пользовательского действия

Повторная отправка **одного и того же пользовательского действия** не должна создавать дубли данных.

Это особенно важно для:

- запроса OTP-кода
- первого входа нового пользователя
- создания обычного матча
- генерации турнирной сетки
- внесения результата турнирного матча
- изменения роли пользователя
- деактивации пользователя

## 3. Глобальные правила нормализации и текстовых полей

## 3.1 Email

Перед валидацией email необходимо:

- удалить пробелы по краям
- привести к нижнему регистру

После нормализации email:

- должен иметь валидный email-формат
- должен сравниваться case-insensitive
- должен быть уникальным case-insensitive

## 3.2 `display_name`

Для `display_name` действуют правила:

- поле обязательно
- после `trim` значение не может быть пустым
- значение не может состоять только из пробелов
- значение не должно превышать лимит схемы
- если имя не было явно задано, система может сгенерировать безопасное значение на основе email

## 3.3 `competition.title`

Для названия турнира действуют правила:

- поле обязательно
- после `trim` значение не может быть пустым
- значение не может состоять только из пробелов
- значение не должно превышать лимит схемы
- raw HTML не должен интерпретироваться как разметка в UI

## 4. Валидация аутентификации и OTP-кода

## 4.1 Запрос кода по email

При запросе одноразового кода система обязана проверить:

- email передан
- email имеет валидный формат после нормализации

Если email невалиден:

- код не отправляется
- сессия не создается
- пользователь не проходит дальше

## 4.2 Ограничение доменов

В Release 1 вход не должен ограничиваться корпоративными доменами.

Система должна:

- принимать любой корректный email
- не использовать allowlist корпоративных доменов как критерий доступа
- не отклонять email только потому, что он не относится к корпоративному домену

Недопустимо:

- требовать корпоративный email для входа
- отклонять запрос OTP только из-за домена email
- показывать пользователю ошибки про "рабочую" или "корпоративную" почту, если email корректен

## 4.3 Создание OTP-кода

Для OTP-кода действуют обязательные правила:

- код является одноразовым
- TTL кода — **10 минут**
- повторная отправка не должна быть доступна чаще одного раза в **30 секунд** для одного email
- для одного активного кода допускается не более **5 попыток ввода**
- после превышения лимита попыток текущий код становится недействительным
- после успешного входа использованный код становится недействительным
- после истечения TTL код становится недействительным

## 4.4 Поведение при повторном запросе OTP

Если пользователь запрашивает новый OTP-код:

- предыдущий активный код для этого email должен быть немедленно инвалидирован
- только последний выданный код считается действительным

Это правило обязательно, чтобы избежать ситуации, когда одновременно валидны несколько кодов.

## 4.5 Валидация одноразового кода подтверждения

Для Release 1 обязательны следующие правила:

- код подтверждения является одноразовым
- код подтверждения действует **10 минут** с момента отправки
- после успешного входа использованный код повторно использовать нельзя
- повторная отправка кода не должна быть доступна чаще одного раза в **30 секунд**
- для одного кода должно быть доступно не более **5 попыток ввода**
- после превышения допустимого количества попыток текущий код должен считаться недействительным
- просроченный код не должен позволять войти в систему

## 4.6 Валидация входа Release 1

В Release 1 вход выполняется по `email + одноразовый код подтверждения`.

### Правила валидации email

Система должна:

- принимать любой корректный email
- не ограничивать вход корпоративными доменами
- удалять случайные пробелы по краям перед проверкой
- сравнивать email в нормализованном виде
- не считать домен email критерием доступа сам по себе

Недопустимо:

- отклонять email только потому, что он не корпоративный
- использовать проверки по списку корпоративных доменов
- показывать пользователю ошибки про "рабочую почту" или "корпоративную почту", если продуктовая логика этого не требует

## 4.7 Проверка деактивированного пользователя

Если пользователь был деактивирован администратором, система должна:

- не позволять ему войти в приложение
- не создавать для него новую сессию
- не допускать доступ к защищенным маршрутам
- показывать понятную ошибку доступа

Подходящая формулировка:

- `Ваш доступ отключен. Обратитесь к администратору`

## 5. Валидация профиля и участника активности

## 5.1 Профиль

Профиль не должен создаваться, если:

- отсутствует `auth_user_id`
- отсутствует email
- email не уникален после нормализации

## 5.2 Participant

Для `participants` обязательны проверки:

- `profile_id` существует
- `activity_type_id` существует
- нельзя создать два `participants` для одной пары `(profile_id, activity_type_id)`
- participant должен создаваться только для допустимой активности
- participant не должен дублироваться при повторном первом входе

## 5.3 Активность Release 1

Для Release 1 пользовательский flow должен работать только для `table_tennis`.

Если через обход UI в запрос подставлена другая активность:

- сервер должен отклонить такое действие там, где Release 1 не предполагает работу с другой активностью в UI-сценарии

## 6. Валидация рейтинга

## 6.1 Рейтинг нельзя редактировать вручную через обычный UI

`rating`, `matches_played`, `wins`, `losses` не должны изменяться произвольным пользовательским вводом через обычный пользовательский интерфейс Release 1.

## 6.2 Неотрицательные значения

Система должна гарантировать, что:

- `rating` остается корректным числом
- `matches_played >= 0`
- `wins >= 0`
- `losses >= 0`

## 6.3 Обновление рейтинга должно быть атомарным

Сохранение результата матча и пересчет рейтинга должны быть частью одной согласованной операции.

Недопустимы состояния:

- матч сохранен, а рейтинг не обновлен
- рейтинг обновлен, а матч не сохранен
- рейтинг одного участника обновлен, а второго — нет

## 6.4 Защита от двойного пересчета

Один завершенный матч должен влиять на Elo только один раз.

Недопустимы ситуации, когда:

- один и тот же обычный матч обновляет рейтинг дважды
- один и тот же турнирный матч обновляет рейтинг дважды
- повторный submit приводит к повторному Elo update

---

## 7. Валидация обычных матчей

## 7.1 Базовые проверки

При создании обычного матча сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- пользователь имеет роль `player`, `organizer` или `admin`
- пользователь является одним из участников матча
- оба участника существуют
- оба участника активны
- оба участника относятся к одной и той же активности
- активность матча совпадает с активностью участников

## 7.2 Матч с самим собой

Матч с самим собой запрещен.

Обязательные проверки:

- `participant1_id != participant2_id`
- в UI нельзя позволять выбрать себя же как второго участника
- на сервере запрос с одинаковыми участниками должен отклоняться

## 7.3 Проверка формата матча

Допустимые форматы:

- `BO1`
- `BO3`
- `BO5`

Недопустимо:

- пустой формат
- любой формат вне allowlist

## 7.4 Проверка числовых значений счета

Для `participant1_score` и `participant2_score` обязательны проверки:

- значение присутствует
- значение является числом
- значение является целым числом
- значение неотрицательное
- значение не является строкой с буквами
- значение не является дробным числом
- значение не является `NaN`
- значение не является `Infinity`

Система должна отвергать:

- `-1`
- `-5`
- `1.5`
- `2,5`
- `abc`
- пустое значение

## 7.5 Проверка результата по формату

Допустимые итоговые результаты:

### Для `BO1`
- `1:0`
- `0:1`

### Для `BO3`
- `2:0`
- `2:1`
- `0:2`
- `1:2`

### Для `BO5`
- `3:0`
- `3:1`
- `3:2`
- `0:3`
- `1:3`
- `2:3`

Любой другой итоговый счет должен быть отклонен.

## 7.6 Ничья запрещена

В Release 1 ничья не поддерживается.

Система должна отвергать:

- `1:1`
- `2:2`
- `3:3`
- любой другой счет, не имеющий одного победителя

## 7.7 Проверка победителя

`winner_participant_id` обязателен.

Сервер обязан проверить:

- победитель указан
- победитель является одним из двух участников
- победитель соответствует итоговому счету

Недопустимы состояния:

- указан победитель, не участвующий в матче
- победитель не соответствует счету
- матч сохранен без победителя

## 7.8 Время матча

Если используется `played_at`, система должна проверить:

- значение имеет корректный datetime-формат
- дата не лежит в явно невозможном формате

Для Release 1 рекомендуется не принимать очевидно будущую дату матча, если продуктовый flow подразумевает сохранение уже сыгранного матча.

## 7.9 Защита от повторного submit при создании матча

Система не должна создавать дубли обычного матча из-за:

- двойного клика по кнопке
- повторной отправки формы браузером
- повторной доставки одного и того же запроса клиентом
- гонки между UI и сетью

Обязательное требование:

- одно пользовательское действие создания матча должно приводить максимум к одной записи матча и одному обновлению Elo

Допустимые способы реализации:

- блокировка повторной отправки в UI
- server-side idempotency
- transaction guard
- комбинация этих способов

## 7.10 Завершенный матч

После сохранения завершенного обычного матча:

- матч не должен редактироваться через UI
- матч не должен удаляться через UI
- формат завершенного матча не должен изменяться через UI

## 8. Валидация турниров

## 8.1 Создание турнира

При создании турнира сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- роль пользователя — `organizer` или `admin`
- название турнира передано
- название турнира не пусто после `trim`
- активность указана корректно
- `format = single_elimination`
- `match_format` входит в allowlist `BO1 / BO3 / BO5`

## 8.2 Владение турниром

Для любых write-действий над конкретным турниром сервер обязан проверить:

- если пользователь `organizer`, он должен быть владельцем турнира
- если пользователь `admin`, ограничение владения не применяется
- `player` не может выполнять write-операции над турниром

## 8.3 Просмотр турниров

Все авторизованные пользователи могут:

- видеть список турниров
- открывать страницу турнира

Но права на просмотр не дают права на управление.

## 8.4 Добавление участников в турнир

При добавлении участника в турнир сервер обязан проверить:

- турнир существует
- турнир находится в статусе `draft`
- пользователь имеет право управлять этим турниром
- participant существует
- participant активен
- participant относится к той же активности, что и турнир
- participant еще не добавлен в этот турнир

Система должна отвергать:

- добавление одного и того же participant дважды
- добавление участника после старта турнира
- добавление участника другой активности
- добавление неактивного участника

## 8.5 Удаление участника из турнира

Удаление участника допустимо только если:

- турнир находится в статусе `draft`
- пользователь имеет право управлять этим турниром
- удаляемый participant действительно входит в турнир

После перехода турнира в `in_progress` удаление участника должно быть запрещено.

## 8.6 Генерация сетки

Генерация сетки допустима только если:

- турнир существует
- турнир находится в статусе `draft`
- пользователь имеет право управлять этим турниром
- у турнира достаточно участников для старта
- сетка еще не была сгенерирована

Для Release 1 рекомендуется считать минимально допустимым количеством участников **2**.

Недопустимо:

- генерировать сетку дважды
- генерировать сетку для `completed` или `cancelled` турнира
- генерировать сетку без участников
- генерировать сетку с неконсистентными слотами

## 8.7 `bye`

Для матчей с `resolution_type = bye` обязательны правила:

- должен существовать только один реальный участник
- победитель должен совпадать с этим участником
- ручной ввод результата для `bye` не требуется
- Elo для `bye` не пересчитывается

Система не должна ожидать ручного заполнения счета для `bye`-матча как для обычного сыгранного матча.

## 8.8 Ввод результата турнирного матча

При сохранении результата турнирного матча сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- роль пользователя — `organizer` или `admin`
- если пользователь `organizer`, он управляет именно этим турниром
- турнир находится в статусе `in_progress`
- match существует
- match не завершен ранее
- `resolution_type = played`, если это обычный сыгранный турнирный матч
- оба участника присутствуют
- счет задан
- счет целочисленный
- счет неотрицательный
- счет соответствует `competition.match_format`
- победитель указан
- победитель является одним из участников
- ничья отсутствует

## 8.9 Защита от повторной записи результата турнирного матча

Система не должна позволять:

- дважды завершить один и тот же турнирный матч
- дважды продвинуть одного и того же победителя
- дважды обновить Elo для одного и того же турнирного матча

Обязательное требование:

- сохранение результата матча, продвижение победителя и обновление Elo должны происходить как одна согласованная операция

## 8.10 Гонки при внесении результата

Если `organizer` и `admin` одновременно пытаются сохранить результат одного и того же турнирного матча:

- система должна гарантировать, что матч не будет завершен двумя разными результатами
- должен быть принят только один согласованный итоговый результат

## 8.11 Завершение турнира

Завершение турнира допустимо только если:

- турнир существует
- турнир находится в статусе `in_progress`
- пользователь имеет право управлять этим турниром
- определен финальный победитель турнира
- сетка не содержит незавершенных обязательных матчей

Недопустимо:

- завершить турнир без победителя
- завершить турнир дважды
- завершить турнир до корректного завершения bracket flow

## 8.12 Отмена турнира

С точки зрения валидации Release 1 отмена турнира должна быть безопасной по отношению к рейтингу.

Если rollback Elo не реализован, UI-отмена должна быть разрешена только пока в турнире нет завершенных матчей, уже повлиявших на рейтинг.

Иначе возникает риск несогласованных данных.

## 9. Валидация административных действий

## 9.1 Изменение роли пользователя

При изменении роли сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- пользователь имеет роль `admin`
- целевая роль входит в allowlist:
  - `player`
  - `organizer`
  - `admin`

Недопустимо:

- установить роль вне allowlist
- позволить не-admin менять роли

## 9.2 Деактивация пользователя

При деактивации сервер обязан проверить:

- действие выполняет `admin`
- пользователь существует
- пользователь еще не деактивирован
- деактивация не нарушает правило последнего active admin

## 9.3 Активация пользователя

При активации сервер обязан проверить:

- действие выполняет `admin`
- пользователь существует
- пользователь действительно был неактивен

## 9.4 Защита последнего active admin

Система должна блокировать:

- понижение роли последнего активного `admin`
- деактивацию последнего активного `admin`

Дополнительное правило:

- если `admin` пытается понизить или деактивировать самого себя, это допустимо только если после операции в системе остается хотя бы один другой активный `admin`

---

## 10. Валидация доступа и маршрутов

## 10.1 Доступ без авторизации

Без авторизации допустима только:

- страница входа

Прямая попытка открыть внутренний маршрут без авторизации должна приводить к блокировке доступа.

## 10.2 Доступ деактивированного пользователя

Если пользователь деактивирован:

- внутренние страницы не должны работать как для активного пользователя
- write-действия должны быть заблокированы
- сервер не должен выполнять команды такого пользователя

## 10.3 Прямой вызов запрещенных действий

Если пользователь пытается вызвать запрещенное действие не через UI, а напрямую:

- действие должно быть отклонено сервером

Это обязательно для:

- admin actions
- organizer-only tournament actions
- match creation
- tournament result reporting
- role changes
- deactivation

---

## 11. Anti-duplicate и anti-race правила

## 11.1 Общий принцип

Критичные write-операции должны быть защищены от дублей и гонок запросов.

## 11.2 Операции, для которых это обязательно

- запрос OTP-кода
- первый вход нового пользователя
- создание обычного матча
- генерация турнирной сетки
- внесение результата турнирного матча
- изменение роли пользователя
- деактивация пользователя

## 11.3 Обязательное поведение

Одно пользовательское действие не должно приводить к:

- двум профилям
- двум participants для одной активности
- двум rankings для одного participant
- двум одинаковым матчам из одного submit
- двойному Elo update
- двойной генерации одной и той же сетки
- двойному продвижению победителя

---

## 12. Правила UI-валидации

UI не является источником истины, но должен предотвращать очевидные ошибки пользователя.

Для Release 1 UI должен:

- не позволять выбрать себя как второго участника матча
- не позволять сохранить матч без формата
- не позволять сохранить матч без счета
- не позволять сохранить матч с явно невалидным счетом
- не позволять сохранить матч без победителя
- блокировать кнопку submit на время запроса
- не показывать действия, на которые у пользователя нет прав
- не показывать `organizer` действия управления чужим турниром
- не показывать `player` organizer/admin actions

## Пользовательские сообщения об ошибках для auth flow

В Release 1 сообщения об ошибках должны быть короткими, понятными и нейтральными.

Подходящие формулировки:

- `Введите корректный email`
- `Введите код подтверждения`
- `Неверный код`
- `Срок действия кода истек`
- `Запросите новый код`
- `Слишком много попыток. Запросите новый код`
- `Ваш доступ отключен. Обратитесь к администратору`

Недопустимо использовать формулировки:

- `Используйте рабочую почту`
- `Доступ только для корпоративной почты`
- `Домен не поддерживается`

если текущая продуктовая логика Release 1 этого не требует

## 13. Критичные негативные сценарии, которые должны быть заблокированы

Система обязана блокировать как минимум следующие сценарии:

Ввод невалидного email
2. Использование просроченного OTP-кода
3. Использование уже использованного OTP-кода
4. Превышение 5 попыток ввода одного OTP-кода
5. Повторную отправку OTP чаще чем раз в 30 секунд
6. Создание дублирующего `profile` при гонке первого входа
7. Создание матча с самим собой
8. Создание матча между двумя другими пользователями
9. Ввод отрицательных значений счета
10. Ввод дробных значений счета
11. Ввод строковых значений вместо числа
12. Сохранение ничьей
13. Сохранение матча без победителя
14. Сохранение матча со счетом, не соответствующим `BO1 / BO3 / BO5`
15. Повторный submit, создающий дубли матчей
16. Добавление одного participant в турнир дважды
17. Добавление participant другой активности в турнир
18. Добавление participant после старта турнира
19. Повторную генерацию сетки
20. Внесение результата уже завершенного турнирного матча
21. Повторное Elo-обновление для одного и того же матча
22. Управление чужим турниром пользователем с ролью `organizer`
23. Доступ `player` к organizer/admin write-actions
24. Изменение роли пользователем без `admin`
25. Деактивацию последнего active admin
26. Понижение последнего active admin
27. Выполнение новых действий деактивированным пользователем
28. пускать в систему только по факту существования email без подтверждения OTP
29. использовать просроченный OTP-код
30. использовать уже примененный OTP-код повторно
31. продолжать принимать код после превышения лимита попыток
32. создавать второй `profile` для того же email
33. разрешать вход деактивированному пользователю
34. отклонять email только из-за не-корпоративного домена

## 14. Краткий итог

`VALIDATION_RULES.md` фиксирует не продуктовый scope, а **правила защиты системы от невалидного ввода, дублей, гонок и запрещенных состояний**.

Главные принципы:

- UI помогает, но сервер обязателен как источник истины
- критичные инварианты должны защищаться на уровне данных
- одно пользовательское действие не должно порождать дублей
- невалидные значения не должны попадать в доменную модель
- все критичные write-операции должны быть защищены от обхода UI
