# RBAC.md — SideQuest Release 1

## 1. Назначение документа

Этот документ описывает **ролевую модель доступа** проекта **SideQuest Release 1**.

RBAC в рамках Release 1 определяет:

- какие роли существуют в системе
- как назначаются роли
- какие действия разрешены каждой роли
- какие действия запрещены каждой роли
- какие серверные проверки обязательны
- какие правила считаются фиксированными и не подлежат свободной интерпретации

Этот документ является главным источником истины по ролям и правам доступа.

Если есть конфликт между этим документом и реализацией, приоритет имеет этот документ.  
Если есть конфликт между этим документом и `PRD.md`, приоритет имеет `PRD.md`.

---

## 2. Общая модель ролей Release 1

### 2.1 Поддерживаемые роли

В системе существуют только три роли:

- `player`
- `organizer`
- `admin`

Других ролей в Release 1 не существует.

### 2.2 Количество ролей у пользователя

В Release 1 у пользователя всегда **ровно одна роль**.

Нельзя:

- назначать пользователю несколько ролей одновременно
- хранить массив ролей у пользователя
- реализовывать переключение ролей
- позволять пользователю выбирать активную роль

### 2.3 Назначение роли новому пользователю

При первом успешном входе нового пользователя система должна:

- создать профиль пользователя
- автоматически назначить роль `player`

Это обязательное правило Release 1.

### 2.4 Изменение ролей

Роли `organizer` и `admin` назначаются только пользователем с ролью `admin`.

Пользователь не может:

- самостоятельно повысить свою роль
- запросом к API поменять себе роль
- переключить интерфейс в режим `organizer` или `admin`, если роль не выдана

## 3. Базовые правила доступа

### 3.1 Общий принцип

Любое действие в системе должно проверяться на двух уровнях:

1. на уровне интерфейса
2. на серверной стороне

### 3.2 Что это означает practically

Система должна:

- скрывать недоступные действия и кнопки в UI
- дополнительно проверять право на действие на сервере
- не полагаться только на скрытие кнопок

### 3.3 Базовое правило авторизации

Все страницы приложения, кроме страницы входа, требуют авторизации.

Без авторизации доступна только:

- страница входа

### 3.4 Источник роли и статуса пользователя

Источник истины для роли и активности пользователя:

- таблица `profiles`

Для проверки доступа используются как минимум:

- `role`
- `is_active`

### 3.5 Правило входа в Release 1

В Release 1 вход выполняется по `email + одноразовый код подтверждения`.

Обязательные правила:

- пользователь может использовать любой корректный email
- email не ограничивается корпоративными доменами
- успешный вход требует подтверждения одноразовым кодом
- при первом успешном входе автоматически создается профиль пользователя
- новому пользователю автоматически назначается роль `player`

В Release 1 не требуется whitelist email и не требуется отдельное предварительное одобрение email для входа.

### 3.6 Деактивированный пользователь

Если пользователь деактивирован:

- он не должен выполнять новые действия в системе
- он не должен иметь доступ к внутренним маршрутам как активный пользователь
- его исторические данные должны сохраняться

## 4. Описание ролей

### 4.1 Роль `player`

`player` — базовая роль обычного пользователя системы.

`player` может:

- пользоваться системой как участник
- видеть собственные данные
- видеть общий рейтинг
- видеть список турниров
- видеть страницу турнира
- видеть список обычных матчей
- создавать обычные матчи только со своим участием

`player` не может:

- создавать турниры
- управлять турнирами
- вводить результаты турнирных матчей
- управлять пользователями
- менять роли
- выполнять административные действия

### 4.2 Роль `organizer`

`organizer` — пользователь, который может организовывать и вести турниры.

`organizer`:

- имеет все права `player`
- может создавать турниры
- может управлять только собственными турнирами
- может добавлять участников в собственный турнир до старта
- может удалять участников из собственного турнира до старта
- может генерировать сетку собственного турнира
- может вводить результаты матчей собственного турнира
- может завершать собственный турнир
- может отменять собственный турнир

`organizer` не может:

- управлять чужими турнирами
- менять роли пользователей
- активировать пользователей
- деактивировать пользователей
- выполнять административные действия вне турнирной зоны ответственности

### 4.3 Обязательные условия доступа

Чтобы пользователь мог выполнять авторизованные действия, должны выполняться условия:

- пользователь успешно аутентифицирован через `email + OTP`
- для пользователя существует профиль или он создается при первом успешном входе
- профиль пользователя активен (`is_active = true`)
роль пользователя определена и допустима

### 4.4 Поведение для деактивированного пользователя

Если пользователь деактивирован:

- он не должен иметь доступ к приложению после ввода email и кода
- он не должен попадать на внутренние маршруты как активный пользователь
- он не должен выполнять новые действия в системе
- система должна обрабатывать этот случай явно и безопасно

Исторические данные такого пользователя при этом сохраняются.

### 4.5 Первый вход нового пользователя

Если пользователь впервые успешно проходит вход через `email + OTP`, система должна:

- создать профиль пользователя
- назначить роль `player`
- пометить пользователя как активного

Пользователь не выбирает роль сам и не проходит отдельную ручную регистрацию.

## 5. Доступ по разделам приложения

### 5.1 Страница входа

Доступ:

- без роли
- без авторизации

Разрешено:

- открыть страницу входа
- ввести любой корректный email
- получить одноразовый код подтверждения
- ввести одноразовый код подтверждения

### 5.2 Главная страница

Доступ:

- `player`
- `organizer`
- `admin`

Условия:

- пользователь авторизован
- профиль активен

### 5.3 Профиль

Доступ:

- `player`
- `organizer`
- `admin`

Разрешено:

- открыть собственный профиль
- видеть имя
- видеть email
- видеть роль
- видеть рейтинг
- видеть последние матчи
- видеть участие в турнирах

### 5.4 Рейтинг

Доступ:

- `player`
- `organizer`
- `admin`

Разрешено:

- открыть страницу рейтинга
- видеть рейтинг участников
- видеть место, имя, Elo, количество матчей

### 5.5 Обычные матчи

Доступ на просмотр:

- `player`
- `organizer`
- `admin`

Доступ на создание:

- `player`
- `organizer`
- `admin`

Специальное ограничение:

Пользователь любой из этих ролей может создать обычный матч **только со своим участием**.

Это означает:

- автор записи должен быть одним из участников матча
- нельзя создать матч между двумя другими пользователями

### 5.6 Турниры — просмотр

Доступ:

- `player`
- `organizer`
- `admin`

Разрешено:

- открыть список турниров
- открыть страницу турнира
- просматривать состав участников
- просматривать статус турнира
- просматривать турнирную сетку

### 5.7 Турниры — создание

Доступ:

- `organizer`
- `admin`

Запрещено для:

- `player`

Разрешено:

- создать турнир
- указать название
- указать активность
- выбрать формат турнирных матчей
- сохранить турнир в статусе `draft`

### 5.8 Турниры — управление

#### Управление собственным турниром

Доступ:

- `organizer`
- `admin`

#### Управление чужим турниром

Доступ:

- `admin`

Запрещено для:

- `organizer`
- `player`

## 6. Правило владения турнирами

В Release 1 доступ к турнирам делится на:

- доступ к просмотру
- доступ к управлению

### 6.1 Доступ к просмотру турниров

Все авторизованные пользователи с ролями:

- `player`
- `organizer`
- `admin`

могут:

- просматривать список турниров
- открывать страницу турнира
- просматривать состав участников
- просматривать статус турнира
просматривать турнирную сетку


### 6.2 Доступ к управлению турнирами

Пользователь с ролью `organizer` может управлять только теми турнирами, которые создал сам.

Владение турниром определяется по полю:

- `competitions.created_by_profile_id`

### 6.3 Для роли `organizer` разрешено только для собственных турниров

- редактировать турнир
- добавлять участников
- удалять участников до старта
- генерировать сетку
- вносить результаты турнирных матчей
- завершать турнир
- отменять турнир

### 6.4 Для роли `organizer` запрещено

- управлять турнирами, созданными другим организатором
- изменять состав участников чужого турнира
- генерировать сетку чужого турнира
- вводить результаты матчей чужого турнира
- завершать или отменять чужой турнир

### 6.5 Для роли `admin`

Пользователь с ролью `admin` может:

- просматривать любые турниры
- управлять любыми турнирами независимо от автора создания

## 8. Обязательные серверные проверки

Для каждого действия, меняющего данные, сервер должен явно проверять права.

### 8.1 Проверки для всех write-действий

Перед выполнением любого write-действия нужно проверить:

- пользователь авторизован
- профиль пользователя существует
- профиль активен
роль пользователя разрешает действие

## 8.1.1 Проверки для входа

При обработке входа сервер обязан проверить:

- email имеет корректный формат
- одноразовый код существует и еще действителен
- одноразовый код не был уже использован
- количество попыток ввода кода не превышено
- если профиль пользователя уже существует, он не деактивирован
- если профиль пользователя не существует, он может быть создан после успешного подтверждения входа

### 8.2 Проверки для обычного матча

При создании обычного матча сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- у пользователя есть роль `player`, `organizer` или `admin`
- пользователь является одним из участников матча
- участники различны
- победитель указан корректно
- матч не является ничьей

### 8.3 Проверки для турниров

При создании и ведении турниров сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- роль — `organizer` или `admin`

Для write-операций над конкретным турниром дополнительно:

- если роль пользователя `organizer`, он должен быть владельцем турнира
- если роль пользователя `admin`, ограничение владения не применяется

### 8.4 Проверки для admin-действий

При изменении роли или активации/деактивации пользователя сервер обязан проверить:

- пользователь авторизован
- пользователь активен
- роль — только `admin`

### 8.5 Проверка последнего active admin

При попытке:

- понизить роль пользователя `admin`
- деактивировать пользователя `admin`

сервер обязан проверить, не является ли этот пользователь последним активным администратором системы.

Если это последний активный `admin`, действие должно быть запрещено.

## 9. Правила назначения и изменения ролей

## 9.1 Автоматическое назначение

При первом успешном входе через `email + OTP`:

- новый пользователь получает только `player`

Это правило не зависит от домена email.

### 9.2 Ручное изменение роли

Изменить роль пользователя может только `admin`.

### 9.3 Допустимые переходы ролей

В Release 1 допустимы любые переходы между:

- `player`
- `organizer`
- `admin`

если действие выполняет `admin` и не нарушается правило защиты последнего active admin.

### 9.4 Что нельзя делать

Нельзя:

- позволять `player` менять свою роль
- позволять `organizer` менять свою роль
- позволять `organizer` повышать других пользователей
- хранить историю ролей как отдельную обязательную сущность Release 1
- давать пользователю выбор роли на экране входа

---

## 10. Правила деактивации пользователя

### 10.1 Что означает деактивация

Деактивация означает:

- пользователь считается существующим
- исторические данные сохраняются
- пользователь перестает быть активным участником системы

### 10.2 Кто может деактивировать

Только `admin`.

## 10.3 Что происходит после деактивации

После деактивации:

- пользователь не должен входить в систему
- пользователь не должен создавать новые матчи
- пользователь не должен создавать турниры
- пользователь не должен управлять турнирами
- пользователь не должен администрировать других пользователей
- исторические записи, связанные с этим пользователем, не должны разрушаться

### 10.4 Защита последнего active admin

Система не должна позволять деактивировать последнего активного пользователя с ролью `admin`.

---

## 11. Правила UI по ролям

### 11.1 Общий принцип

UI должен отражать права роли, но не заменять серверную проверку.

### 11.2 Обязательное поведение UI

Для каждой роли UI должен:

- показывать только доступные действия
- скрывать недоступные кнопки и ссылки
- не предлагать маршрут, который гарантированно запрещен текущей роли

### 11.3 Что не должен делать UI

UI не должен:

- показывать `player` кнопку создания турнира
- показывать `player` кнопку ввода результата турнирного матча
- показывать `organizer` административный раздел пользователей
- показывать переключатель ролей
- показывать пользователю возможность выбрать роль при входе
- показывать `organizer` действия управления чужим турниром

---

## 12. Фиксированные правила Release 1

Ниже — правила, которые нельзя трактовать свободно:

1. В Release 1 существуют только роли `player`, `organizer`, `admin`.
2. У пользователя в Release 1 только одна роль.
3. Новый пользователь автоматически получает роль `player`.
4. Пользователь не выбирает роль при входе.
5. Пользователь не переключает роль внутри системы.
6. Только `admin` может менять роли пользователей.
7. Только `admin` может активировать и деактивировать пользователей.
8. Последний активный `admin` не может быть понижен или деактивирован.
9. Вход в Release 1 выполняется по `email + одноразовый код подтверждения`.
10. Пользователь может использовать любой корректный email.
11. Новый пользователь при первом успешном входе автоматически получает роль `player`.
12. Деактивированный пользователь не может войти в систему и не может выполнять новые действия.
13. Все страницы, кроме страницы входа, требуют авторизации.
14. Для всех write-действий нужна серверная проверка роли.
15. `player` может создавать обычный матч только со своим участием.
16. `player` не может создавать турниры.
17. `player` не может вводить результаты турнирных матчей.
18. `organizer` может создавать турниры.
19. `organizer` может управлять только собственными турнирами.
20. `organizer` не может управлять чужими турнирами.
21. `organizer` не может управлять ролями пользователей.
22. `admin` может выполнять все действия `organizer`.
23. `admin` может управлять пользователями.
24. Все авторизованные пользователи могут просматривать список турниров и страницу турнира.
25. UI должен скрывать недоступные действия, но этого недостаточно без серверной проверки.

## 13. Критерии корректной реализации RBAC

RBAC Release 1 реализован корректно, если:

1. новый пользователь всегда создается как `player`
2. вход выполняется по `email + OTP`
3. пользователь может использовать любой корректный email
4. пользователь не может выбрать роль при входе
5. `player` не видит и не может вызвать organizer/admin actions
6. `organizer` не видит и не может вызвать admin actions
7. `admin` имеет доступ к user management
8. деактивированный пользователь не может войти и не может выполнять новые действия
9. доступ к чувствительным операциям проверяется на серверной стороне
10. не используется multi-role модель
11. не используется permission matrix, выходящая за scope Release 1
12. поведение ролей соответствует `PRD.md`, `DOMAIN_MODEL.md`, `TASKS.md`, `AGENTS.md`

## 14. Краткий итог модели доступа

Release 1 использует простую и жестко зафиксированную модель:

- один пользователь
- одна роль
- один набор прав на основе роли
- обязательная серверная проверка
- минимально необходимый набор прав без избыточной сложности

Итоговая логика:

- `player` — участвует
- `organizer` — организует собственные турниры
- `admin` — управляет пользователями и имеет максимальные права Release 1
