# TASKS.md — SideQuest Release 1

## 1. Назначение документа

Этот документ задает **порядок, границы и критерии выполнения задач** для Release 1 проекта **SideQuest**.

Цель документа:

- разбить реализацию Release 1 на небольшие и однозначные этапы
- исключить хаотичное добавление функциональности
- не допустить переписывания уже работающих частей проекта без необходимости
- помочь Codex выполнять задачи последовательно и предсказуемо

Этот документ не заменяет другие документы проекта и должен использоваться вместе с ними.

---

## 2. Приоритет и порядок чтения документов

Перед выполнением любой задачи сначала нужно открыть `DOC_INDEX.md`, а затем читать документы в порядке, указанном там.

Для практической работы над задачами использовать документы в следующем порядке:

1. `DOC_INDEX.md`
2. `PRD.md`
3. `DOMAIN_MODEL.md`
4. `RBAC.md`
5. `TASKS.md`
6. `TECH_STACK.md`
7. `VALIDATION_RULES.md`
8. `AGENTS.md`

Если документы конфликтуют, применять приоритеты из `DOC_INDEX.md`.

## 3. Общие правила выполнения задач

### 3.1 Общий принцип

Каждая задача должна:

- реализовывать **один законченный кусок функциональности**
- иметь четкие границы
- не выходить за scope текущего этапа
не затрагивать unrelated части проекта

### 3.2 Что запрещено при выполнении любой задачи

Нельзя без явного запроса:

- менять стек проекта
- заменять ORM
- заменять способ работы с БД
- переписывать уже работающую архитектуру
- массово переименовывать сущности
- менять уже согласованные таблицы и контракты
- добавлять out-of-scope функции
- делать большой рефакторинг “заодно”
- трогать unrelated файлы ради “улучшения”

### 3.3 Что обязательно перед началом любой задачи

Перед выполнением любой задачи нужно:

1. прочитать `DOC_INDEX.md`
2. прочитать документы, относящиеся к текущей задаче, в правильном порядке
3. определить, какой именно этап проекта реализуется
4. определить, какие файлы относятся к текущей задаче
5. определить, какие файлы не должны изменяться

### 3.4 Что обязательно после выполнения любой задачи

После выполнения любой задачи нужно:

- кратко перечислить, что было реализовано
- перечислить измененные файлы
- указать, какие проверки были выполнены
- честно указать, что не было сделано, если это оставалось вне scope
не утверждать, что реализовано больше, чем реально сделано

## Правило activity scope для Release 1

При реализации экранов, серверных выборок и пользовательских списков Release 1 необходимо считать, что все основные данные работают **в контексте текущей активности**, даже если в UI пока нет отдельного фильтра или переключателя активности.

В Release 1 текущая активность фиксирована и равна:

- `table_tennis`
- **Настольный теннис**

### Что это означает для реализации

Во всех задачах Release 1 нужно исходить из того, что:

- рейтинг строится **по текущей активности**
- последние матчи строятся **по текущей активности**
- список обычных матчей строится **по текущей активности**
- турниры строятся **по текущей активности**
- участие пользователя и его статистика интерпретируются **по текущей активности**

### Обязательное правило для задач Release 1

Даже если в интерфейсе Release 1 нет видимого activity selector, реализация не должна трактовать основные экраны как глобальные экраны "по всей платформе".

Недопустимо в задачах Release 1:

- строить глобальный рейтинг по всем активностям сразу
- смешивать матчи разных активностей в одном пользовательском списке
- смешивать турниры разных активностей в одном пользовательском списке
- писать серверные выборки без activity scope там, где экран логически относится к текущей активности

### Следствие для реализации экранов

При выполнении задач, связанных с экранами:

- `Главная`
- `Профиль`
- `Рейтинг`
- `Матчи`
- `Турниры`
- `Страница турнира`

Codex должен считать, что данные этих экранов уже в Release 1 выбираются в рамках `table_tennis`, даже если фильтр активности пока не отображается в UI.

### Следствие для будущего расширения

Реализация Release 1 должна позволять в будущем добавить выбор активности без переписывания базовой логики экранов.

Поэтому уже в Release 1 нужно строить реализацию так, чтобы переход к нескольким активностям означал добавление нового activity context, а не полную переделку существующих экранов и выборок.
